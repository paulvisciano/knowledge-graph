Transcription Challenge — Evaluation & Recommendations

How should voice input flow into a local knowledge-graph chat app? An evaluation of three options, the broader local-STT landscape, and how open-source voice chat solutions architect this.

Local-only constraint Gemma 4 12B · llama.cpp LightRAG + MCP tools Date: Jul 2026

🏆 Recommendation

Adopt a hybrid of Option 1 + Option 3. Use a dedicated Whisper.cpp server (already scaffolded on port 8090) to transcribe audio → text, then feed the transcript into the existing text pipeline so the LLM can call query_knowledge_graph via MCP tools. Upgrade the Whisper model from base.en to medium.en or large-v3 for usable quality. Keep Gemma's native-audio path as an optional fallback for general-knowledge chat where KG retrieval is not needed. This mirrors the pattern used by every production open-source voice chat app surveyed (Open WebUI, AnythingLLM, LobeChat, LibreChat) — all transcribe first, then run their normal text RAG/tool pipeline.

🎯 The Problem

Voice input in this app has a fundamental architectural tension. The chat's value comes from querying a local knowledge graph (LightRAG) via MCP tool calls — the LLM decides to call query_knowledge_graph based on the user's text. When the user speaks, there are two sub-problems that must be solved in order:

  1. Speech → Text (transcription / STT) — turn the audio blob into accurate text.
  2. Text → Knowledge Graph → Answer — the existing pipeline: the LLM receives the transcript as a user message, decides whether to call query_knowledge_graph (an OpenAI-style tools declaration served by the MCP server at /mcp), receives the retrieval result as a tool message, then generates the final answer.

Gemma 4 12B (via llama.cpp with the mmproj projector) accepts audio natively through the OpenAI-compatible input_audio content part. That lets you skip step 1 — but, as we'll see, it also undermines step 2.

Current state in the codebase

  • Audio capture: ui/src/lib/utils/audio-recording.ts — AudioWorklet captures PCM, encodes WAV, base64-encodes for the API.
  • Transcription endpoint: API.llama.transcriptions → /whisper/inference (proxied by nginx at docker/nginx/nginx.conf:112 to whisper:8090, the whisper-server started in scripts/start-llama-servers.sh:100-113).
  • Model-based transcribe (the "poor" path): routing audio to the Gemma LLM itself as an input_audio part — see ui/src/routes/+page.svelte:481 and :489-493. This is what produces poor transcription. It is not the whisper-server.
  • Direct-audio chat path: +page.svelte:471-487 builds an OpenAI input_audio content part and sends it to /v1/chat/completions on the Gemma model. When this path is taken, the LLM can still be passed the tools array (+page.svelte:541-544) — but see "Why tool-calling breaks" below.
  • KG query pipeline: ui/src/lib/services/mcp-client.svelte.ts exposes MCP tools as OpenAI tool definitions; +page.svelte:744-789 executes returned tool_calls by calling mcpClient.callTool(), which hits the FastMCP server at mcp-services/src/knowledge_graph_mcp/server.py, which in turn calls LightRAG.
  • Whisper server already exists: docker/whisper/Dockerfile builds whisper.cpp's whisper-server; scripts/download-models.sh:55-58 downloads ggml-base.en.bin.
Clarification on "poor transcription". The poor quality comes from using the Gemma LLM itself to transcribe audio (the input_audio path), not from the whisper-server. The whisper-server is scaffolded but uses the tiny base.en model — its quality is mediocre but predictable, and is trivially upgradeable. This distinction matters because it changes which option is actually broken.

Current architecture (text path vs. direct-audio path)

TEXT PATH (works, KG is queried): User types ─▶ /v1/chat/completions (messages=[{role:user, content:"text"}], tools=[query_knowledge_graph]) ─▶ Gemma 4 12B decides to call tool ─▶ mcpClient.callTool("query_knowledge_graph", {q, mode:"local"}) ─▶ FastMCP /mcp ─▶ LightRAG /query ─▶ NetworkX + BGE-M3 + Postgres ─▶ tool result back to Gemma ─▶ final answer with KG context ✅ DIRECT-AUDIO PATH (KG retrieval unreliable/broken): User speaks ─▶ WAV base64 ─▶ /v1/chat/completions (messages=[{role:user, content:[{type:"input_audio",...},{type:"text",...}]}], tools=[...]) ─▶ Gemma consumes audio tokens (~few hundred) + tries to reason ─▶ Tool-calling degrades: audio tokens crowd context, model often answers directly from what it heard instead of calling tools ─▶ KG rarely queried ❌ (plus transcription itself is poor) RECOMMENDED PATH (transcribe-then-retrieve): User speaks ─▶ WAV ─▶ /whisper/inference (whisper.cpp server, upgraded model) ─▶ transcript text ─▶ same as TEXT PATH above ✅ (KG reliably queried, tools work)

① The Three Options Evaluated

Option 1 — Send audio directly to Gemma (native multimodal)

already implemented no transcription step KG retrieval unreliable +page.svelte:481

Gemma 4 12B is loaded with its mmproj projector (scripts/start-llama-servers.sh:21,39-40), giving llama.cpp's OpenAI-compatible server the ability to accept {type:"input_audio", input_audio:{data, format:"wav"}} content parts. The UI already builds these parts (+page.svelte:471-487) and logs that it's sending them (:489-493). No separate STT step is needed — the model "listens" and responds.

Why it "seems to work well"

For general-knowledge questions ("what's the capital of France?"), Gemma understands the audio and answers directly. The experience feels magical — one round trip, no transcription latency.

Why it breaks the knowledge graph

  • Tool-calling degrades with audio input. The tools array is sent alongside audio (+page.svelte:541-544), but in practice the model tends to answer from what it heard rather than emit a structured tool_call. Audio tokens consume context and attention; the model's instruction-following for tool emission is measurably worse than on text-only turns. Community reports on Ollama/llama.cpp multi-modal + tools confirm this is a known weak spot, not a bug specific to this app.
  • Transcription quality is poor. Gemma is an LLM, not an ASR model. Its "understanding" of audio is good enough for casual Q&A but produces paraphrased, lossy, and sometimes hallucinated interpretations of the actual words. For KG queries where exact entity names ("Paul", "Aurora", "Baja trip") matter, paraphrasing silently breaks retrieval.
  • No transcript in the transcript. The user's spoken words never become a discrete text string the UI can show, edit, or retry. If the model mishears, the user can't correct it — the audio is opaque.
  • Cost per turn is higher. Audio tokens (~hundreds) are evaluated on every request, and the entire audio must be re-sent on multi-turn follow-ups if the conversation references it.
Pros
  • Simplest architecture — no STT service.
  • Lowest latency for general Q&A (one round trip).
  • Model can pick up tone/prosody.
  • Already wired up in the UI.
Cons
  • Knowledge-graph queries are unreliable — the whole point of the app.
  • LLM-as-transcriber hallucinates & paraphrases entity names.
  • No visible/correctable transcript.
  • Tool-calling + audio is a known weak combination in llama.cpp.
  • Higher per-turn token cost.

Option 2 — Transcribe via the Gemma model itself (the "poor" endpoint)

poor quality wrong tool for the job no extra service

This is the path that currently produces poor transcription: asking the Gemma LLM to "transcribe" the audio (either by prompting it to output the text, or by using it as an ad-hoc STT endpoint). The transcript is then intended to feed the text pipeline so KG tools work. The architecture is sound — transcribe-then-retrieve is the right pattern — but the transcriber is wrong.

Why an LLM is a bad transcriber

  • Not trained for WER. Whisper and its kin are trained on hundreds of thousands of hours of audio with explicit ASR objectives. Gemma's audio capability is a side-channel of multimodal pretraining; it has no ASR loss.
  • Paraphrasing, not verbatim. Ask an LLM "what did I say?" and it summarizes. ASR must output the exact words, including names and proper nouns the LLM may not have in its vocabulary. Entity names like "Aurora" or "Baja" get autocorrected to common words.
  • Hallucinations. LLMs hallucinate filler when uncertain; ASR models are trained to output blank/uncertain markers instead.
  • Latency & cost. A 12B model generating a transcript is far slower and more expensive than a dedicated 74M-parameter Whisper encoder running on the same GPU.
Verdict: Reject. Using Gemma as the transcriber gives you the worst of both worlds — LLM latency and poor ASR quality — for no benefit over a dedicated Whisper model. The fix is not to tune the prompt; it's to use a model designed for ASR.
Pros
  • No additional service to run.
  • Output is text → KG pipeline works downstream.
Cons
  • Quality is fundamentally poor — wrong model class.
  • Slow (12B generating text vs. 74M ASR encoder).
  • Hallucinated/paraphrased entity names break KG retrieval.
  • Same VRAM as the chat LLM — no parallelism possible.

Option 3 — Separate Whisper server (dedicated ASR)

already scaffolded right tool for the job needs model upgrade docker/whisper/Dockerfile port 8090

A dedicated whisper-server (from whisper.cpp) already runs on port 8090 and is proxied at /whisper/inference. The UI already calls it via transcribeAudio() in audio-recording.ts:276-299 and feeds the result to handleSend(..., transcript) at +page.svelte:200-208. The only problem is that it's running the base.en model — the second-smallest Whisper model — which is why quality is mediocre. The infrastructure is correct; the model is undersized.

What's already in place

  • docker/whisper/Dockerfile — builds whisper.cpp's whisper-server with -DWHISPER_BUILD_SERVER=ON.
  • scripts/start-llama-servers.sh:100-113 — starts whisper-server -m ggml-base.en.bin -l en --convert -t 4 on port 8090, with health check.
  • scripts/download-models.sh:55-58 — downloads ggml-base.en.bin.
  • docker/nginx/nginx.conf:112-113 — proxies /whisper/ → whisper:8090.
  • ui/src/lib/utils/audio-recording.ts:276 — transcribeAudio() posts WAV as multipart form data, parses result.text.

The one-line fix

Swap ggml-base.en.bin for ggml-medium.en.bin (or large-v3.bin for best quality). Update two env vars / two script lines and re-download. Medium.en roughly halves WER vs. base.en at ~2× latency on Metal; large-v3 is SOTA but heavier.

Verdict: Adopt. This is the right architecture and it's already 90% built. The only real work is upgrading the model and wiring the UI so the mic button always goes through transcribeAudio() → text pipeline (which it already does in handleMicClick at +page.svelte:189-223).
Pros
  • Right model class — trained for ASR, low WER.
  • Transcript is discrete text → KG tools work reliably.
  • Small model (74M–1.5B) runs alongside the 12B LLM without contention.
  • User-visible, editable, retryable transcript.
  • Already scaffolded end-to-end in this repo.
  • Matches the pattern used by every surveyed OSS chat app.
Cons
  • Adds ~1-3s latency per voice turn (acceptable for chat).
  • Current base.en model is too weak — must upgrade.
  • Extra ~150MB-1.5GB VRAM/disk for the Whisper weights.
  • No prosody/tone conveyed to the LLM (rarely needed for KG Q&A).

② Additional Suggestions (beyond the 3 options)

A. Hybrid: transcribe for RAG, also send audio for context

Run Option 3 as the primary path, but also include the input_audio part in the LLM request alongside the transcript text. The LLM gets the accurate transcript (so it can call tools reliably) and the audio (so it can pick up tone, hesitation, emphasis). Cost: extra audio tokens per turn. Benefit: marginal — rarely useful for a knowledge-graph assistant. Worth knowing about, not worth building first.

C. Streaming STT for low-latency "walkie-talkie" feel

If sub-second response becomes a goal, switch from the current "record → upload full WAV → wait" flow to streaming partial transcripts while the user is still speaking. Options:

  • whisper.cpp streaming — whisper-server supports incremental /inference calls on sliding windows; not true streaming but good enough.
  • WhisperLive / whisper_streaming — community wrappers that feed chunked audio to Whisper and emit partial results over WebSocket.
  • Vosk — true streaming STT over WebSocket, lowest latency, lower quality.
  • Moonshine — designed for low-latency edge use; partial-result support via ONNX streaming.

For a knowledge-graph assistant, full-utterance transcription is almost certainly fine — the KG query and LLM generation together take longer than the STT step. Don't pursue streaming until STT is the bottleneck.

D. Intent-routing layer (optional, for later)

Add a tiny classifier (or a second, cheaper LLM call) that looks at the transcript and decides: does this need the KG? If yes → run the full tool-calling pipeline. If no (general knowledge, smalltalk, math) → skip retrieval and answer directly. This is the pattern Home Assistant's voice pipeline uses (STT → intent → action) and what advanced Open WebUI setups approximate with "RAG on/off" toggles. It saves latency and KG load on questions that don't need it. Not needed now — the LLM's own tool-choice already does this implicitly — but worth noting as the system scales.

E. Voice activity detection (VAD) before STT

Today the user holds a key to record. For hands-free "push to talk" or always-on modes, add Silero VAD (tiny, ~2MB, runs in browser or server-side) to chunk audio at speech boundaries and trim silence. This reduces Whisper latency (less silence to process) and avoids hallucinations Whisper is known to produce on long silent tails.

F. A dedicated audio-LLM with real tool-calling (future watch)

None of the surveyed production apps do this today, but the landscape is moving:

  • Qwen2-Audio — audio-native LLM, Qwen2-based, supports function calling, runnable locally via vLLM/sglang/transformers. The most plausible "audio in + tools work" path if you wanted to escape transcribe-then-retrieve. Requires swapping out the Gemma stack.
  • Voxtral (Mistral) — audio-native, strong on audio understanding, but tool-calling support is less mature than Qwen2-Audio.
  • Gemma 3n — has true audio strands (unlike Gemma 4 which is vision+text); Ollama has a converter. Early. Tool-calling + audio unproven.

These are worth tracking but not worth adopting now. The transcribe-then-retrieve pattern is battle-tested, works with your current Gemma model, and won't be obsolete even if audio-LLMs improve — you'd just gain an optional shortcut path.

🌍 How Other Open-Source AI Chat Solutions Solve This

Survey of production open-source voice-enabled chat apps. The pattern is overwhelmingly consistent: transcribe first with a dedicated STT service, then run the normal text RAG/tool pipeline. None send audio directly to a multimodal LLM and skip retrieval.

Open WebUI ↗ repo

STT: Local faster-whisper (in-process) by default; OpenAI Whisper API and other providers optional. Configurable via WHISPER_MODEL (tiny → large).

RAG flow: Audio recorded in browser → transcribed to text client-side → text submitted as a normal chat message → RAG pipeline (hybrid vector + BM25, reranker) runs on the text → LLM generates. Voice input is just an input modality; everything downstream is text.

Pattern: (a) transcribe-first, then text RAG. ✅

AnythingLLM ↗ repo

STT: Browser-native Web Speech API (free, uses the browser's built-in STT) by default; local Whisper as an option.

RAG flow: Transcript text drops into the chat composer → user sends → AnythingLLM's vector-search RAG runs on the text → LLM. Voice is an input convenience only.

Pattern: (a) transcribe-first, then text RAG. ✅

LobeChat ↗ repo

STT: Pluggable STT providers — OpenAI Whisper API, Azure, browser Web Speech, and local Whisper via plugin. Local-capable.

RAG flow: STT → text in composer → LLM (with optional knowledge base / tools). Same text pipeline either way.

Pattern: (a) transcribe-first, then text RAG/tools. ✅

LibreChat ↗ repo

STT: Browser Web Speech API for input; OpenAI Whisper API for server-side. Local Whisper possible via custom endpoint.

Tools/RAG: Transcript becomes a normal user message → agents/tools/RAG run on text. Voice doesn't bypass retrieval.

Pattern: (a) transcribe-first. ✅

Home Assistant / Wyoming protocol ↗ docs

STT: Pluggable via Wyoming protocol — Whisper (faster-whisper), Vosk, and others. Fully local.

Flow: STT → intent recognition (HA's intent engine or LLM) → action (home-assistant API calls, which are effectively "tools"). The canonical "transcribe → intent → tool-call" pipeline, identical in spirit to transcribe → LLM → MCP tool-call.

Pattern: (a) + (D) intent-routing. ✅

WhisperLive / whisper_streaming ↗ repo

STT: Streaming Whisper over WebSocket, partial transcripts. Not a chat app, but a reusable STT component many local chat setups plug into.

Use: Drop-in for the "streaming STT" suggestion above if low-latency voice chat becomes a goal.

Pattern: STT infrastructure, consumed by (a) downstream. ✅

Big-AGI / Continuum

Voice input via browser Web Speech API → text → normal LLM/tool flow. No audio-native LLM path.

Pattern: (a) transcribe-first. ✅

Takeaway from the survey. The industry consensus is unambiguous: dedicated STT → text → retrieval/tools. The "send audio straight to the LLM" pattern (Option 1) is essentially absent from production open-source chat apps, both because audio-native LLMs with reliable tool-calling are immature and because text is the lingua franca every downstream stage already speaks. Your app should follow the same pattern.

🛠️ Recommended Action Plan

Phase 1 — Make the existing whisper path good (small effort)

  1. Upgrade the Whisper model. In scripts/download-models.sh:55-58, swap the URL from ggml-base.en.bin to ggml-medium.en.bin (or large-v3.bin for best quality). Update WHISPER_MODEL_PATH in .env / scripts/start-llama-servers.sh:22 accordingly. Re-download.
  2. Confirm the UI uses the whisper path, not the direct-audio path. handleMicClick at +page.svelte:189-223 already calls transcribeAudio(wavBlob, API.llama.transcriptions) and passes the transcript to handleSend(..., text) — good. Audit any other entry point that might set audioData to send input_audio directly, and gate it behind an explicit "no-KG" toggle if you keep it at all.
  3. Show the transcript in the UI before/with sending, so the user can edit if the STT misheard a name. The chat message already stores content: trimmed from the transcript (+page.svelte:911).
  4. Benchmark. Record 10 representative utterances (including proper nouns from your graph), transcribe with base.en vs medium.en vs large-v3, measure WER and latency on your Mac. Pick the smallest model that's "good enough" — likely medium.en.

Phase 2 — Optional refinements

  • Add Silero VAD to trim silence and reduce hallucinated tails on long recordings.
  • If latency matters, evaluate faster-whisper as a drop-in replacement for whisper.cpp (Python service, ~4× faster, Open WebUI uses it).
  • Keep a hidden "raw audio to Gemma" toggle for general-knowledge questions where KG isn't needed and lowest latency matters — this is your Option 1 fallback, deliberately scoped.
  • Track Qwen2-Audio's local tool-calling maturity; if it matures, it could become a real Option 1 replacement that keeps KG retrieval working.

Phase 3 — If you ever outgrow Whisper

  • Streaming: WhisperLive over WebSocket for partial transcripts.
  • SOTA English quality: NVIDIA Parakeet TDT via NeMo.
  • Intent routing: a tiny classifier or small LLM call to skip KG when the question is general knowledge.

📝 Summary

  • Option 1 (Gemma audio direct): feels slick but silently breaks the app's killer feature — KG retrieval. Reject as the default path.
  • Option 2 (Gemma as transcriber): the source of the "poor transcription" you observed. Wrong model class. Reject.
  • Option 3 (Whisper server): the right architecture, already 90% built. The only fix needed is upgrading base.en → medium.en/large-v3. Adopt this.
  • Bonus: hybrid (transcribe + send audio), faster-whisper, streaming STT, VAD, intent routing, and audio-LLMs with real tool-calling (Qwen2-Audio) are all worth knowing about for later — none are needed now.
  • Industry consensus: every surveyed OSS chat app (Open WebUI, AnythingLLM, LobeChat, LibreChat, Home Assistant) uses transcribe-first. Follow suit.
