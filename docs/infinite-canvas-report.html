Replacing the 3D Force Graph with an Infinite Canvas

A performance-focused architecture report for the knowledge-graph UI, modeled on edoardolunardi/infinite-canvas.

SvelteKit Three.js 3d-force-graph React Three Fiber (reference) Performance

Executive summary

The site renders its knowledge graph with 3d-force-graph on top of Three.js, building a custom THREE.Object3D per node. Performance is bad for three compounding reasons:

  1. Every node is in the scene at once. There is no virtualization, frustum culling, or distance-based culling. The whole graph (all nodes + all edges) is handed to graph.graphData(buildGraphData()) on every refresh.
  2. Photo nodes use full-resolution images as GPU textures. The /photo/{filename} endpoint serves raw files via FileResponse with no resizing; the client converts them to base64 data URLs and uploads them as textures. A 4–12 MB JPEG becomes a 4–12 MB GPU texture per photo node.
  3. Force simulation + scene.traverse() run continuously while the user pans/zooms. The codebase already carries comments acknowledging that traversing the full scene every frame was "a major source of frame drops on initial load."

The reference repo infinite-canvas solves exactly this with chunk-based rendering + distance-based culling: the infinite 3D space is divided into CHUNK_SIZE cubes, only chunks within RENDER_DISTANCE of the camera are mounted, and each plane fades/skips updates by camera distance. Five planes per chunk, an LRU plane cache, throttled chunk updates during zoom, and a texture cache with on-demand loading keep it smooth at any dataset size.

This report proposes replacing the force-graph view with an infinite-canvas built on the same chunk/culling pattern, adapted to Svelte + the existing Three.js dependency, with a thumbnail image pipeline, a filter/detail system, and a chat-to-canvas bridge so AI-referenced images surface on the canvas in real time.

Current architecture

Stack

  • UI: SvelteKit (ui/, package name "nexus"), Svelte 5 runes, Vite, Tailwind v4.
  • Graph lib: 3d-force-graph + d3-force / d3-zoom / d3-selection.
  • 3D: three 0.185 (already a direct dependency).
  • Backend: FastAPI (api/) over a LightRAG fork (lightrag/) storing a Neo4j-style graph.

Bottlenecks (with evidence)

#BottleneckWhereImpact
1 No culling / no virtualization. All nodes & edges are in the WebGL scene simultaneously. No frustum culling flag, no distance gating, no chunking. buildGraphData() returns the entire graph each refresh. GraphCanvas.svelte buildGraphData, .graphData(buildGraphData()) at 1163/1316/1335/1349/1382/1528/1610 Frame time scales with total node count, not visible count. The scene graph grows unbounded as new nodes stream in over SSE.
2 Full-resolution images as GPU textures. /photo/{filename} returns the raw file (FileResponse, no PIL resize). Client turns it into a base64 data URL, then a THREE.Texture. Only face crops are thumbnailed to 256×256. images.py:437-451; graph.svelte.ts:83-102; GraphCanvas.svelte:603-631 (loadPhotoTexture) VRAM blowup (each multi-MB JPEG → uncompressed texture), decode stalls on the main thread, GC pressure from many Image+dataURL objects.
3 Per-frame scene.traverse() + manual opacity RAF loop. The codebase added "dirty flags" to avoid traversing every frame, which confirms traversing was a measured hotspot. GraphCanvas.svelte:120-125 (comments), opacity RAF loop Competes with the force-graph render loop and zoomToFit camera tween during initial load — the documented cause of initial frame drops.
4 Force simulation keeps running / restarting. Every data merge calls graph.graphData(buildGraphData()) then resumeAnimation(), re-cooking the whole layout. GraphCanvas.svelte:1316-1338, 1610-1637 O(n) physics tick over all nodes on every incremental update; visible "jitter" and CPU spikes during SSE merges.
5 Per-node object creation is expensive. createNodeThreeObject allocates new PlaneGeometry, EdgesGeometry, MeshBasicMaterial, and a 128² CanvasTexture (person) per node. No geometry/material reuse. GraphCanvas.svelte:709-… Allocation + disposal churn as nodes are added/removed; draw calls scale 1:1 with nodes.
6 No image LOD / progressive loading. The first photo fetch loads the full image; there is no small-thumb-then-full-replace pattern like the reference's texture-progress loader. graph.svelte.ts:72-105 Long time-to-first-image on large galleries; bandwidth wasted on off-screen photos.

Reference: infinite-canvas (how it stays fast)

Repo: edoardolunardi/infinite-canvas — React 19 + React Three Fiber + Three.js 0.182, MIT. It explores thousands of artworks on an endless 3D grid while staying smooth. The performance strategy is concentrated in three files:

CHUNK_FADE_MARGIN = 1     // extra ring that fades out
DEPTH_FADE_START  = 140
DEPTH_FADE_END    = 260

  // RENDER_DISTANCE + CHUNK_FADE_MARGIN  → ~125 chunks max

src/infinite-canvas/utils.ts

  • generateChunkPlanesCached — 5 planes/chunk, LRU-cached (MAX_PLANE_CACHE = 256).
  • getChunkUpdateThrottleMs — 500 ms while zooming fast, 400 ms while zooming, 100 ms idle.
  • shouldThrottleUpdate — gates React state updates so setChunks doesn't fire every frame.

src/infinite-canvas/texture-manager.ts

  • Global Map<url, THREE.Texture> cache — each image loaded once.
  • TextureLoader with mipmap + anisotropy + sRGB on load.
  • Per-key callback set so multiple planes sharing a texture all get notified.

src/infinite-canvas/scene.tsx (the loop)

  • useFrame moves camera by velocity (drag/wheel/keys/touch), then computes cx,cy,cz = floor(pos / CHUNK_SIZE).
  • On chunk change, setChunks(CHUNK_OFFSETS.map(...)) mounts only nearby chunks (throttled).
  • Each MediaPlane does its own useFrame fade: opacity target is 1 if dist <= RENDER_DISTANCE, else a fade ramp; mesh.visible=false + depthWrite=false when below INVIS_THRESHOLD.
  • Depth fade (absDepth > DEPTH_FADE_END) hides far planes entirely.
Net effect: only ~125 chunks × 5 planes = ~625 meshes exist at any time, and most are invisible or fading. Dataset size is irrelevant to frame time. Textures load on demand with a global progress bar (useProgress).

Proposed architecture

Replace GraphCanvas.svelte with a new CanvasView.svelte that renders the knowledge graph as an infinite 2.5D canvas (mostly-flat 3D planes with parallax depth), reusing the reference's chunk/culling pattern. Keep Svelte + the existing three dependency — no React, no R3F needed.

Why 2.5D, not the 3D force graph

  • Photos are 2D objects; a flat plane grid maps naturally. The reference proves a near-flat Z with parallax depth reads well and culls cheaply.
  • Removing the force simulation eliminates bottleneck #3 and #4 entirely. Layout becomes a pure function of the graph (see Data model).
  • Chunk coordinates become meaningful (a node lives at a stable (cx,cy,cz)) instead of drifting every time physics re-runs.

Data model & layout

Canvas node (front-end only — no backend change)

type CanvasNode = {
  id: string;                 // KGNode.id
  labels: string[];
  properties: Record<string, unknown>;
  kind: 'photo' | 'person' | 'location' | 'event' | 'concept';
  imageUrl?: string;          // resolved thumbnail URL (LOD)
  fullUrl?: string;           // full-res URL (for overlay)
  // assigned by Layout.ts:
  cellX: number; cellY: number; cellZ: number; // chunk coords
  localX: number; localY: number; localZ: number; // offset within chunk
  width: number; height: number; // plane size
};

Layout strategy (replaces force simulation)

  • Primary dimension = time. Chunk X = ordered ingestion time bucket. New photos naturally scroll in from the right.
  • Y within chunk = cluster. Group by the photo's dominant cluster/person/event from the graph; clusters form horizontal bands. Reuse the existing hubMap grouping logic in GraphCanvas.svelte:92-94.
  • Z = relationship depth. The selected/focused node's neighbors sit at the camera plane; 2-hop nodes drop back in Z (parallax), giving the "focus and get more details" depth feel.
  • Positions are stable (assigned once per node, stored in a Map) so panning back doesn't re-lay-out.

Chunk & culling (ported from the reference)

Port these primitives almost verbatim into TS (no React needed):

From constants.ts

Keep CHUNK_SIZE, RENDER_DISTANCE, CHUNK_FADE_MARGIN, INVIS_THRESHOLD, DEPTH_FADE_*, and the precomputed CHUNK_OFFSETS. Tune CHUNK_SIZE for photo aspect ratios (e.g. 160 so a 4:3 plane fits with margin).

From scene.tsx loop

  • One requestAnimationFrame loop: apply velocity → compute cx,cy,cz → if changed, queue a throttled chunk update.
  • Throttle by zoom speed (port getChunkUpdateThrottleMs): 500/400/100 ms.
  • Mount/unmount chunk THREE.Groups; reuse a pool to avoid GC.

From MediaPlane fade

  • Per-plane: target opacity = dist <= RENDER_DISTANCE ? 1 : fade(dist).
  • Below INVIS_THRESHOLD: mesh.visible=false, material.depthWrite=false.
  • Depth fade hides far planes outright.

From utils.ts

  • generateChunkPlanesCached LRU — but seeded by our Layout.ts assignment, not random.
  • Plane pool: shared 1×1 PlaneGeometry (the reference's PLANE_GEOMETRY constant) scaled per mesh → one geometry for the whole canvas.

Image pipeline (fixes bottleneck #2)

This is the highest-ROI change and is independent of the canvas rewrite — it can land first.

  1. Add a thumbnail endpoint. New GET /api/kg/photo/{filename}?w=256 in api/routes/images.py that returns a PIL-resized JPEG (port the existing face-crop thumbnail((256,256), LANCZOS) pattern). Cache to disk under dist/thumbs/. Serve full-res only on explicit ?w=full.
  2. Two-stage LOD on the canvas. Each NodePlane loads the 256 thumbnail first (instant), then the full image swaps in when the plane is hovered/selected (the detail overlay already needs full-res).
  3. Global TextureCache keyed by URL (port of texture-manager.ts), not by nodeId. Multiple nodes referencing the same image share one texture. Delete the per-node photoTextureCache in GraphCanvas.svelte:598.
  4. Drop data URLs. Stop FileReader.readAsDataURL in graph.svelte.ts:93-97. Serve a real URL and let TextureLoader fetch it — frees main-thread decode time and ~33% memory (base64 overhead).
  5. Lazy load only mounted chunks. Because chunks are culled, off-screen photos are never fetched. Today every photo is fetched eagerly on graph load.

Filter & detail

Filter bar (FilterBar.svelte)

  • Type chips: Photo / Person / Location / Event / Organization / Concept — maps to existing TYPE_COLORS in GraphCanvas.svelte:17-24.
  • Text search: reuse graphStore.filteredNodes logic (already does label + properties + id substring match) — but instead of filtering the data array, set a matchMask per node. Non-matching planes fade to opacity 0.08 and become non-interactive (the reference's fade primitive makes this trivial).
  • Facets: date range (from EXIF date_taken, already in the graph), person (from personImages set), location (GPS cluster). Derived from node properties client-side.
  • Focus mode: a filter that also re-centers the camera on the matching nodes' bounding box (replaces graphCanvas.fitView() at GraphView.svelte:516).

Detail overlay (NodeOverlay.svelte)

  • Evolve PhotoOverlay.svelte to be node-type-aware (it already computes neighbor persons via edges at lines 35–44 — generalize to all neighbor types).
  • Triggered on plane click/raycast. Full-res image loads on open (the LOD second stage). Metadata panel shows EXIF, people, locations, events from graph properties.
  • "Show on canvas" action that flies the camera to the node's chunk and highlights its neighborhood (the requested "get more details on a given image" experience).

Chat → canvas bridge (the new feature)

Today the chat extracts image filenames from an assistant's ### References section (extract-reference-images.ts) and renders them as an inline carousel in the chat bubble (+page.svelte:499-514). The user wants referenced images to come up on the canvas as the AI mentions them.

Proposed flow

assistant stream → on '### References' detected:
  refNames = extractReferenceImages(content)
  for name in refNames:
    node = graphStore.findPhotoByFilename(name)   // new helper
    canvas.highlight(node.id)                     // sets opacity 1 + cyan ring
    canvas.flyTo(node.id)                          // camera tween to chunk
    chat.pushRefCard(node.id)                      // chat shows a compact "📷 shown on canvas" card
  • Streaming-aware: parse references incrementally as tokens arrive (the regex in extract-reference-images.ts is already line-oriented and safe to run on partial content).
  • Camera tween: reuse the reference's velocity model — set targetVel toward the target chunk and let the existing damping bring the camera there smoothly (no jarring zoomToFit).
  • Persistent highlight: referenced planes get a selection ring (the codebase already builds EdgesGeometry rings at GraphCanvas.svelte:745-748 — reuse for referenced images).
  • Two-way: clicking a plane that the AI referenced also scrolls the chat to the matching message (link by messageId ↔ nodeId).

Effort & risk

Estimated effort

  • Phase 0 — image pipeline: ~1 day (port the existing face-crop PIL pattern).
  • Phase 1 — canvas skeleton: ~2–3 days (port constants/utils/texture-manager to TS; Svelte RAF loop).
  • Phase 2 — layout & selection: ~2 days.
  • Phase 3 — filter & detail: ~1–2 days (reuse filteredNodes + PhotoOverlay).
  • Phase 4 — chat bridge: ~1 day (small, additive).
  • Phase 5 — cut over: ~0.5 day.

Total: ~1.5–2 weeks of focused work.

Key risks

  • Stable layout under streaming SSE. Force graph absorbs new nodes by re-cooking; a static layout must assign cells to new nodes without shifting existing ones. Mitigation: time-bucket X coordinate + append-only.
  • Raycasting through many planes. Use Three's Raycaster.intersectObjects against the mounted-chunk meshes only (~hundreds), not all nodes. Cheap.
  • VRAM from many visible photos. Mitigated by thumbnails + chunk culling; the reference caps ~625 meshes.
  • Mobile. The reference supports touch; reuse use-swipe.ts/use-breakpoint.ts already in the repo.
