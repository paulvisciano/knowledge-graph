🟢 Existing Strengths

Dedicated Mobile Shell with Tablet Support
+layout.svelte branches on $isMobile || $isTablet to render the nexus-mobile-shell with top bar + bottom tab bar + bottom-sheet drawers. Tablets no longer fall through to the desktop layout.
Responsive Chat Input Bar
chat-input-bar switches to position: fixed on mobile with bottom: calc(56px + env(safe-area-inset-bottom, 0px) + 8px) — clears the tab bar and respects safe-area insets across devices.
Full-Screen Chat Overlay
Chat overlay panel expands to inset: 0 on mobile via @media (max-width: 768px), taking the full screen.
Bottom-Sheet Drawers with Swipe-to-Dismiss
NavDrawer, HistoryPanel, and SettingsDrawer render as bottom sheets on mobile with createSwipeHandler wiring onSwipeDown to close — a standard mobile interaction pattern.
44px Tap Targets on Mobile
All drawer buttons use h-11 w-11 (44px) on mobile per Apple HIG / Material guidelines. Chat input bar buttons use h-10 (40px). GraphControls zoom buttons are 44×44px on mobile.
Global Touch Optimizations
-webkit-tap-highlight-color: transparent, touch-action: manipulation, overscroll-behavior: none all set globally in app.css to reduce mobile jank. viewport-fit=cover enables safe-area support.
WCAG-Compliant Viewport
app.html viewport meta allows pinch-to-zoom (width=device-width, initial-scale=1, viewport-fit=cover) — meets WCAG 2.1 SC 1.4.4 (Resize Text).
Node Detail as Bottom Card
node-detail-overlay repositions from right-side panel (desktop) to bottom-positioned card with max-height: 50dvh on mobile.
Mobile-Aware Graph Rendering
GraphCanvas uses nodeRelSize($isMobile ? 12 : 6) for larger node tap targets and constrains tooltip to max-width: min(250px, 80vw). GraphControls move to top: 60px on mobile to avoid overlap with node detail.
No Mobile UI Framework Dependency
Framework7 was removed (commit c27e696). The mobile UI is now built with native Svelte components + Tailwind — lighter bundle, no framework conflict risk, full control over touch behavior.

🟡 Medium-Priority Issues

1. Chat Not Surfaced in Mobile Tab Bar (Discoverability)

Medium Effort: M MobileTabBar.svelte +page.svelte

Problem

Chat is accessible on mobile — the floating chat input bar at the bottom of the graph view (chat-input-bar in +page.svelte line 1464) expands into a full-screen chat overlay panel (chatExpanded state) when the user interacts with it. However:

  • MobileTabBar.svelte only exposes Graph, Ingest, and Activity — no Chat tab
  • Chat is only discoverable by visiting the Graph tab and finding the floating input bar
  • Users on the Ingest or Activity tabs have no visible path to chat
  • The mobile top bar has History and Settings buttons but no Chat shortcut

This is a discoverability issue, not an access issue. It persists from the original audit.

Recommended Changes

  • Add a Chat tab to MobileTabBar.svelte with icon chat and set activeTab to a new chat tab id
  • Or add a Chat button to the mobile top bar next to History/Settings
  • When the Chat tab is active, render the full-screen chat overlay directly
  • Update TAB_ID type and +page.svelte tab routing
// MobileTabBar.svelte — add Chat tab
const tabs = [
  { id: 'graph', icon: 'graph', label: 'Graph' },
  { id: 'chat', icon: 'chat', label: 'Chat' },   // new
  { id: 'ingestion', icon: 'upload', label: 'Ingest' },
  { id: 'activity', icon: 'activity', label: 'Activity' },
];

2. Ingestion Panel Document Table Overflow Risk

Medium Effort: M IngestionPanel.svelte

Problem

IngestionPanel.svelte renders a document table (<table class="w-full text-left text-xs">, line 302) with multiple columns. On mobile:

  • No @media rules or mobile-specific CSS exist in the component
  • A multi-column table on a 375px screen will horizontally overflow or compress columns to unreadable widths
  • The overflow-y-auto wrapper (line 296) handles vertical scroll but not horizontal overflow
  • No overflow-x-auto on the table container

Recommended Changes

  • Wrap the table in a overflow-x-auto container so it scrolls horizontally rather than breaking layout
  • On mobile, switch from table to a stacked card layout (one document per card) using $isMobile
  • Hide low-priority columns (e.g. file size, status detail) on mobile, show only name + status
  • Reduce pageSize from 20 to 10 on mobile to limit scroll length
// IngestionPanel.svelte — horizontal scroll fallback
<div class="flex-1 overflow-y-auto">
  <div class="overflow-x-auto">  // wrap table
    <table class="w-full min-w-[500px] text-left text-xs"> ... </table>
  </div>
</div>

3. Graph Search Results Not Mobile-Optimized — ✅ Resolved

Resolved Effort: M GraphSearch.svelte

Problem

GraphSearch.svelte rendered search results with no mobile-specific layout, a fixed PAGE_SIZE = 50 (long scroll on mobile), and badge fonts at 10px (below the mobile readability threshold).

🔵 Low-Priority Issues

4. Small Font Sizes Below Mobile Readability Threshold

Low Effort: S +page.svelte app.css

Problem

Several text elements use font sizes at or below 10px, which is below the 12px mobile readability threshold recommended by WCAG 1.4.4:

  • tabbar-label in MobileTabBar: font-size: 10px
  • text-[10px] used in chat message timestamps and ingestion panel
  • text-[11px] used in chat mode toggle and system prompt banner meta
  • text-[12px] in system prompt content — borderline acceptable

Recommended Changes

  • Increase tab bar labels to 11px minimum
  • Bump timestamps and metadata text to 12px on mobile
  • Add a mobile-specific font-size floor in app.css: @media (max-width: 768px) { body { font-size: 15px; } }
  • Use clamp() for critical text: font-size: clamp(12px, 3.5vw, 14px)

5. Missing Haptic Feedback on Touch Interactions

Low Effort: S MobileTabBar.svelte NavDrawer.svelte

Problem

The mobile UI relies on visual feedback (active states, color changes) but provides no haptic feedback for key touch interactions. Mobile users expect subtle haptic confirmation when tapping tab bar items, opening drawers, or selecting graph nodes.

Recommended Changes

  • Add navigator.vibrate(10) on tab bar item taps (guarded by 'vibrate' in navigator)
  • Add haptic feedback when opening/closing NavDrawer, HistoryPanel, SettingsDrawer
  • Add a light vibration when a graph node is selected on mobile
  • Create a shared haptic.ts utility to centralize the logic
// lib/utils/haptic.ts
export function haptic(pattern: number | number[] = 10) {
  if (typeof navigator !== 'undefined' && 'vibrate' in navigator) {
    navigator.vibrate(pattern);
  }
}

// Usage in MobileTabBar.svelte
onclick={() => { haptic(10); activeTab.set(tab.id); }}

6. AudioPlayer Play Button Below 44px Tap Target

Low Effort: S AudioPlayer.svelte

Problem

The new AudioPlayer.svelte component (added in commit a2fb1d8) renders a play/pause button with h-8 w-8 (32×32px) — below the 44px minimum tap target recommended by Apple HIG and Material Design. This is the only interactive control in the player and is used for every voice message in chat.

  • Play/pause button: h-8 w-8 = 32px (line 183)
  • The waveform scrubber is 40px tall (h-10) — acceptable
  • No mobile-specific override exists

Recommended Changes

  • Increase play button to h-10 w-10 (40px) or h-11 w-11 (44px) on mobile
  • Apply the same md: responsive pattern used by NavDrawer/HistoryPanel: h-11 w-11 md:h-8 md:w-8
  • Increase the SVG icon from 14px to 18px to match the larger button

7. PhotoOverlay Card Uses Fixed Pixel Dimensions

Low Effort: S PhotoOverlay.svelte

Problem

The new PhotoOverlay.svelte photo card (tracking photo nodes in the 3D graph) uses hardcoded pixel dimensions that could overflow narrow screens:

  • Thumbnail: width: 140px; height: 105px (line ~620 in style)
  • Info panel: min-width: 160px; max-width: 220px
  • Total card width ≈ 300–360px — on a 320px screen (iPhone SE), the card overflows
  • No @media rules or mobile-specific sizing
  • The card is positioned with absolute left/top from graph coordinates — could clip at screen edges

Recommended Changes

  • Add a mobile media query constraining the card: max-width: calc(100vw - 32px)
  • Reduce thumbnail to width: 100px on mobile
  • Clamp cardX / cardY in tick() so the card never leaves the viewport
  • Consider position: fixed centered overlay on mobile instead of tracking the 3D node
// PhotoOverlay.svelte — mobile constraint
@media (max-width: 768px) {
  .photo-card-thumb-btn img { width: 100px; height: 75px; }
  .photo-card-info { min-width: 120px; max-width: calc(100vw - 140px); }
}

✅ Resolved Issues (Archive)

These issues from the original v1 audit have been fixed. Kept for reference.

R1. Viewport Meta Disables Pinch-to-Zoom — ✅ Resolved

Resolved Effort: S app.html

Problem

The viewport meta tag in app.html previously set maximum-scale=1, user-scalable=no, which prevented users from pinch-to-zoom. This violated WCAG 2.1 Success Criterion 1.4.4 (Resize Text) and was considered an accessibility blocker.

Applied Fix

✅ Fixed in ui/src/app.html — viewport meta now: width=device-width, initial-scale=1, viewport-fit=cover
  • Removed maximum-scale=1 and user-scalable=no
  • Kept viewport-fit=cover for safe-area support
  • touch-action: manipulation in app.css prevents double-tap zoom jank

R2. Sub-44px Tap Targets on Multiple Components — ✅ Resolved

Resolved Effort: M NavDrawer, HistoryPanel, SettingsDrawer, ChatSidebar

Problem

Multiple components used button sizes at or below 28px (h-7 w-7), below the 44px minimum tap target recommended by Apple HIG and Material Design.

Applied Fix

✅ Fixed in commit 28f2dfa — all drawer buttons now use h-11 w-11 md:h-7 md:w-7
  • NavDrawer, HistoryPanel, SettingsDrawer: icon buttons upsized to 44px on mobile
  • ChatSidebar new/collapse buttons: upsized to 44px on mobile
  • Chat input bar buttons in +page.svelte: upsized to h-10 (40px)
// Responsive pattern applied across drawers
class="flex h-11 w-11 md:h-7 md:w-7 shrink-0 items-center
       justify-center rounded-lg border ..."

R3. Swipe-to-Dismiss Not Wired to Mobile Drawers — ✅ Resolved

Resolved Effort: M NavDrawer, HistoryPanel, SettingsDrawer

Problem

use-swipe.ts implemented a swipe gesture handler but it was never used by the mobile bottom sheets. Users could only dismiss drawers by tapping the backdrop.

Applied Fix

✅ Fixed in commit 28f2dfa — createSwipeHandler wired to all three drawers
  • NavDrawer: onSwipeDown: () => close()
  • HistoryPanel: onSwipeDown: () => close()
  • SettingsDrawer: onSwipeDown: () => close()

R4. Tablet Layout Falls Through to Desktop Shell — ✅ Resolved

Resolved Effort: S +layout.svelte

Problem

+layout.svelte only branched on $isMobile. Tablets (768–1023px) fell through to the desktop layout with no mobile tab bar or bottom-sheet drawers.

Applied Fix

✅ Fixed in commit c27e696 — layout now branches on $isMobile || $isTablet
  • Imported isTablet from use-breakpoint
  • Tablets now get the mobile shell (top bar, tab bar, bottom sheets)
  • Safe-area handling applies to tablets in landscape with notches

R5. Chat Input Bar Overlap with Mobile Tab Bar — ✅ Resolved

Resolved Effort: S +page.svelte

Problem

The chat input bar used a fixed bottom: 64px on mobile, which didn't account for env(safe-area-inset-bottom) and could be hidden behind the iOS keyboard.

Applied Fix

✅ Fixed — now uses safe-area-aware calc positioning
  • bottom: calc(56px + env(safe-area-inset-bottom, 0px) + 8px)
  • position: fixed on mobile (was absolute)
  • max-width: 100% with padding: 0 8px

R6. 3D Force Graph Touch Interaction Controls — 🟡 Partially Resolved

Partial Effort: L GraphCanvas.svelte, GraphControls.svelte

Problem

The 3D force graph (3d-force-graph) had tooltip overflow, small node tap targets, and zoom controls that overlapped the node detail card on mobile.

Applied Fixes

🟡 Partially fixed in commit 28f2dfa — tooltip, node size, and control position addressed
  • ✅ Tooltip: max-width: min(250px, 80vw) prevents overflow
  • ✅ Node tap targets: nodeRelSize($isMobile ? 12 : 6) doubles node size on mobile
  • ✅ Zoom controls: repositioned to top: 60px on mobile, upsized to 44×44px

Remaining Work

  • Pinch-to-zoom sensitivity and two-finger pan could be tuned via enableZoomInteraction / enablePanInteraction
  • Node hover tooltip is pointer-events: none — touch devices need a tap-to-show pattern (currently shows on hover which doesn't translate to touch reliably)
