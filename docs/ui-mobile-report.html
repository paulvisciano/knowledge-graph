✅ Existing Mobile Strengths

The app has a surprisingly strong mobile foundation. These patterns should be preserved and built upon:

Separate Mobile Shell
+layout.svelte branches on $isMobile to render a completely separate top-bar + content + tab-bar layout instead of the desktop sidebar shell.
Breakpoint System
use-breakpoint.ts provides reactive isMobile, isTablet, isDesktop stores with resize + orientationchange listeners.
Bottom Tab Bar
MobileTabBar.svelte has a fixed bottom bar with Graph / Ingest / Activity tabs, safe-area padding, blur background, and active-state glow indicator.
Bottom-Sheet Drawers
NavDrawer, HistoryPanel, and SettingsDrawer all render as bottom sheets on mobile (mobile-sheet-bottom) with drag handles and backdrop dismiss.
Swipe Gesture Handler
use-swipe.ts implements touch-based swipe detection with velocity thresholding — ready to wire up to drawer dismiss gestures.
Safe-Area Insets
env(safe-area-inset-*) used in MobileTabBar, mobile topbar, and viewport meta has viewport-fit=cover for notched devices.
Responsive Node Detail
node-detail-overlay repositions from right-side panel (desktop) to bottom-positioned card with max-height: 50dvh on mobile.
Full-Screen Chat Overlay
Chat overlay panel expands to inset: 0 on mobile via @media (max-width: 768px), taking the full screen.
Touch Optimizations
-webkit-tap-highlight-color: transparent, touch-action: manipulation, overscroll-behavior: none all set globally to reduce mobile jank.

🔴 Critical Issues — All Resolved

1. Chat Not Surfaced in Mobile Tab Bar (Discoverability)

Medium Effort: M MobileTabBar.svelte +page.svelte

Problem

Chat is accessible on mobile — the floating chat input bar at the bottom of the graph view (chat-input-bar in +page.svelte line 1456) expands into a full-screen chat overlay panel (chatExpanded state) when the user interacts with it. However:

  • MobileTabBar.svelte only exposes Graph, Ingest, and Activity — no Chat tab
  • Chat is only discoverable by visiting the Graph tab and finding the floating input bar
  • Users on the Ingest or Activity tabs have no visible path to chat
  • The mobile top bar has History and Settings buttons but no Chat shortcut

This is a discoverability issue, not an access issue.

Recommended Changes

  • Add a Chat tab to MobileTabBar.svelte with icon message or chat
  • Map the chat tab to the existing chat overlay panel (set chatExpanded = true and switch to graph view if needed)
  • Alternatively, add a Chat button to the mobile top bar next to History and Settings
  • Ensure the activeTab store and TabId type in stores/ui support a 'chat' value
// MobileTabBar.svelte — add chat tab
const tabs = [
  { id: 'graph', icon: 'graph', label: 'Graph' },
  { id: 'chat', icon: 'message', label: 'Chat' },  // NEW
  { id: 'ingestion', icon: 'upload', label: 'Ingest' },
  { id: 'activity', icon: 'activity', label: 'Activity' },
];

2. Viewport Meta Disables Pinch-to-Zoom (Accessibility Violation) — ✅ RESOLVED

Resolved Effort: S app.html

Problem

The viewport meta tag in app.html previously set maximum-scale=1, user-scalable=no, which prevented users from pinch-to-zoom. This violated WCAG 2.1 Success Criterion 1.4.4 (Resize Text) and was considered an accessibility blocker. 

// app.html line 5 — before (fixed):
<meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no, viewport-fit=cover" />

Applied Changes

  • Removed maximum-scale=1 and user-scalable=no from the viewport meta tag
  • Kept viewport-fit=cover for safe-area support
  • The existing touch-action: manipulation in app.css already prevents double-tap zoom jank
// app.html — now:
<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover" />

✅ Fixed in ui/src/app.html.

🟠 High-Priority Issues

3. 3D Force Graph Lacks Touch Interaction Controls

High Effort: L GraphCanvas.svelte GraphControls.svelte

Problem

GraphCanvas.svelte uses 3d-force-graph (Three.js-based 3D visualization). While the library supports touch gestures natively, the mobile experience has issues:

  • Node tooltip (max-width: 250px) could overflow narrow screens
  • The zoom controls (GraphControls.svelte) are repositioned to bottom: 140px on mobile — but this may overlap with the node detail card (bottom: 120px)
  • No pinch-to-zoom hint or gesture guidance for first-time mobile users
  • 3D rendering is GPU-intensive — no fallback to a 2D graph layout on low-end mobile devices
  • Node label text sizes aren't scaled for mobile readability at zoom-out levels

Recommended Changes

  • Add $isMobile detection to GraphCanvas.svelte and reduce node/label density on mobile
  • Constrain tooltip max-width to min(250px, 80vw) to prevent overflow
  • Add a 2D force-directed layout fallback for mobile (e.g. d3-force with SVG/Canvas rendering)
  • Move zoom controls to top: 60px on mobile to avoid overlap with node detail card
  • Increase touch hit area for node selection — use nodeRelSize to boost tap target size on mobile
  • Add a one-time "Pinch to zoom · Drag to rotate" tooltip overlay on first mobile visit
// GraphCanvas.svelte — mobile-aware node sizing
const nodeRelSize = $isMobile ? 12 : 6;
graph.nodeRelSize(nodeRelSize)
     .nodeLabel(node => `
       <div style="max-width: min(250px, 80vw); ...">${node.labels[0]}</div>
     `);

4. Sub-44px Tap Targets on Multiple Components

High Effort: M NavDrawer.svelte ChatSidebar.svelte +page.svelte

Problem

Apple's Human Interface Guidelines and Material Design both recommend a minimum 44×44px tap target for touch interfaces. Many interactive elements in the app are smaller:

  • ChatSidebar.svelte: h-7 w-7 (28px) buttons for new conversation and collapse toggle
  • +page.svelte: h-9 w-9 (36px) chat history button and attachment buttons
  • NavDrawer.svelte: close button and nav-icon-btn are small
  • HistoryPanel.svelte: close button is h-7 w-7 (28px)
  • Chat message timestamps and action buttons use text-[10px] with h-3 w-3 (12px) icons

Recommended Changes

  • Use Tailwind responsive prefixes: h-7 w-7 md:h-7 md:w-7 h-11 w-11 pattern — larger on mobile, compact on desktop
  • Or use a utility class: .min-tap-target { min-width: 44px; min-height: 44px; }
  • Add invisible padding around small buttons to expand the hit area without changing visual size
  • Increase icon-only buttons to h-11 w-11 (44px) on mobile, keep h-7/h-8/h-9 on desktop
// Pattern: larger on mobile, compact on desktop
class="flex h-11 w-11 md:h-7 md:w-7 items-center justify-center rounded-lg ..."

// Or via CSS in app.css
@media (max-width: 768px) {
  button:not(.no-grow) {
    min-width: 44px;
    min-height: 44px;
  }
}

5. Swipe-to-Dismiss Not Wired to Mobile Drawers

High Effort: M use-swipe.ts NavDrawer.svelte HistoryPanel.svelte SettingsDrawer.svelte

Problem

use-swipe.ts implements a full swipe gesture handler with directional callbacks and velocity thresholding — but it doesn't appear to be used by any of the mobile bottom sheets. Mobile users can only dismiss drawers by tapping the backdrop or pressing the close button or Escape key (which doesn't exist on mobile keyboards).

Bottom sheets should support swipe-down-to-dismiss — a standard mobile interaction pattern that users expect.

Recommended Changes

  • Import and use createSwipeHandler in NavDrawer, HistoryPanel, and SettingsDrawer
  • Attach onSwipeDown to the bottom-sheet panel element to call close()
  • Add visual feedback during swipe — translateY the sheet following the finger (requires onTouchMove tracking, currently a no-op)
  • Consider adding onSwipeRight to NavDrawer for right-edge swipe-to-open when closed
// NavDrawer.svelte — wire up swipe-to-dismiss
import { createSwipeHandler } from '$lib/composables/use-swipe';

let panelEl: HTMLDivElement;

$effect(() => {
  if (!$isMobile || !$navDrawerOpen || !panelEl) return;
  const handler = createSwipeHandler({
    element: panelEl,
    onSwipeDown: () => close(),
    threshold: 60,
  });
  return () => handler.destroy();
});

6. Tablet Layout Falls Through to Desktop Shell

High Effort: M +layout.svelte

Problem

+layout.svelte only branches on $isMobile. Tablets (768px–1023px) fall through to the desktop nexus-shell layout, which uses a fixed floating menu trigger and a sidebar drawer. On a tablet in portrait orientation, this results in:

  • No mobile tab bar — the tablet gets the desktop floating menu button instead
  • No bottom-sheet drawers — the tablet gets the desktop side-panel drawers
  • No safe-area handling (tablets in landscape with notches)
  • The isTablet and isMobileOrTablet stores exist in use-breakpoint.ts but are never used in the layout

Recommended Changes

  • Branch the layout on $isMobileOrTablet instead of just $isMobile
  • Or add a three-way branch: mobile (<768px) → mobile shell, tablet (768–1023px) → adaptive shell, desktop (≥1024px) → desktop shell
  • For tablet: use the mobile tab bar but allow side-by-side panels in landscape
  • Consider lowering the mobile breakpoint to 640px if tablet portrait should use desktop layout
// +layout.svelte — three-way branch
{#if $isMobile}
  <div class="nexus-mobile-shell"> ... </div>
{:else if $isTablet}
  <div class="nexus-tablet-shell"> ... </div>
{:else}
  <div class="nexus-shell"> ... </div>
{/if}

🟡 Medium-Priority Issues

7. Ingestion Panel Document Table Overflow Risk

Medium Effort: M IngestionPanel.svelte

Problem

IngestionPanel.svelte renders a document list with file paths, status badges, timestamps, and action buttons. On narrow mobile screens, these multi-column rows can overflow horizontally or compress to unreadable widths. The component has no mobile-specific styles or responsive breakpoints.

  • File paths (potentially long) have no truncation or wrapping strategy
  • Status badges + action buttons compete for horizontal space
  • No pagination controls visible at pageSize = 20 — could be a long scroll on mobile
  • The drag-and-drop upload zone ("dragging" state) has no mobile equivalent — mobile users can't drag files

Recommended Changes

  • Truncate file paths with truncate / overflow-hidden text-ellipsis whitespace-nowrap
  • Stack status and actions vertically on mobile using flex-col md:flex-row
  • Increase row padding on mobile for easier tapping: py-3 md:py-2
  • Replace drag-and-drop zone with a prominent "Tap to Upload" button on mobile (use $isMobile)
  • Reduce pageSize to 10 on mobile to reduce scroll fatigue
  • Use a card-based layout on mobile instead of a row-based table

8. Chat Input Bar Overlap with Mobile Tab Bar

Medium Effort: S +page.svelte

Problem

The chat input bar is positioned at bottom: 64px on mobile to clear the 56px tab bar + safe area. However:

  • The 64px offset doesn't account for env(safe-area-inset-bottom) which varies by device
  • When the keyboard opens on iOS, 100dvh shrinks but the input bar's fixed bottom: 64px may leave it hidden behind the keyboard
  • The attachment menu and model dropdown open upward from the input bar — on mobile these could be clipped by the top of the screen

Recommended Changes

  • Use bottom: calc(56px + env(safe-area-inset-bottom, 0px) + 8px) instead of fixed 64px
  • Use the Visual Viewport API to handle keyboard: window.visualViewport.addEventListener('resize')
  • Or use position: sticky within a scrolling container instead of position: absolute
  • Ensure attachment menu and model dropdown use placement: top with max-height constraints on mobile
// +page.svelte — safe-area-aware positioning
@media (max-width: 768px) {
  .chat-input-bar {
    bottom: calc(56px + env(safe-area-inset-bottom, 0px) + 8px);
    max-width: 100%;
    padding: 0 8px;
  }
}

9. Graph Search Results Not Mobile-Optimized

Medium Effort: M GraphSearch.svelte

Problem

GraphSearch.svelte renders search results with entity type badges, descriptions, and related entity lists. On mobile:

  • Result cards may be too compact for touch interaction
  • No mobile-specific layout — the same desktop rendering is used
  • Search input may be too small (pageSize = 50 results could create a very long scroll)
  • Entity type color badges use small font sizes that are hard to read on mobile

Recommended Changes

  • Add mobile-specific card layout: larger padding, single-column, bigger fonts
  • Reduce PAGE_SIZE to 20 on mobile
  • Increase result card tap target — make the entire card clickable, not just the label
  • Use $isMobile to switch from multi-column to single-column result grid
  • Add a sticky search input at the top with blur-backdrop on mobile scroll

🔵 Low-Priority Issues

10. Small Font Sizes Below Mobile Readability Threshold

Low Effort: S +page.svelte app.css

Problem

Several text elements use font sizes at or below 10px, which is below the recommended minimum of 12px for mobile readability:

  • Chat timestamps: text-[10px]
  • Thinking indicator: text-[10px], text-[11px]
  • Mobile tab bar labels: font-size: 10px
  • Various status badges and meta text: text-xs (12px)

Recommended Changes

  • Increase mobile tab bar labels to font-size: 11px minimum
  • Use text-[11px] md:text-[10px] pattern for timestamps — larger on mobile
  • Ensure body text is never below 14px on mobile for main content areas
  • Add a CSS custom property for minimum mobile font size and enforce via linter

11. Missing Haptic Feedback for Touch Interactions

Low Effort: S MobileTabBar.svelte NavDrawer.svelte

Problem

The mobile UI relies on visual feedback (active states, color changes) but provides no haptic feedback for key touch interactions. Mobile users expect subtle haptic confirmation when tapping tab bar items, opening drawers, or selecting graph nodes.

Recommended Changes

  • Add navigator.vibrate(10) on tab bar item taps (guarded by 'vibrate' in navigator)
  • Add haptic feedback when opening/closing NavDrawer, HistoryPanel, SettingsDrawer
  • Add a light vibration when a graph node is selected on mobile
  • Create a shared haptic.ts utility to centralize the logic
// lib/utils/haptic.ts
export function haptic(pattern: number | number[] = 10) {
  if (typeof navigator !== 'undefined' && 'vibrate' in navigator) {
    navigator.vibrate(pattern);
  }
}

// Usage in MobileTabBar.svelte
onclick={() => { haptic(10); activeTab.set(tab.id); }}
