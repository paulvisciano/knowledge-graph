1. Whisper variants for local deployment

The three production-grade Whisper runtimes all load the same OpenAI Whisper weights (tiny → large-v3, plus the 809M-param large-v3-turbo), so WER is essentially identical across them. The decision is about the runtime/kernel, not the model.

1.3 WhisperX, distil-whisper — what they add

  • WhisperX (m-bain/whisperX) wraps faster-whisper and adds three things the base runtimes lack: Silero-VAD preprocessing (avoids hallucinating on silence), word-level forced alignment via wav2vec2 (drops timestamp drift from ~1s to <100ms), and speaker diarization via pyannote.audio. v3.8.5 is current. VRAM stays under 8 GB for large-v2 with beam_size=5. Best when output format matters — subtitles, meeting transcripts, anything that needs "who said what, when."
  • distil-whisper (HuggingFace, distil-large-v3) is a distilled Whisper trained by Intel. ~2× faster than large-v3 with a small WER penalty; faster-whisper and whisper.cpp both load the distil checkpoints directly. Use when you want large-class quality at medium-class cost.

1.4 How to serve them

Option A — embedded library. faster-whisper is a Python pip install; you call model.transcribe(audio, language="en") from your FastAPI route. Lowest operational overhead.

Option B — whisper.cpp server. whisper.cpp ships an examples/server HTTP endpoint. Crucially, run it as a persistent backend, not as a per-file CLI — Allen Kuo's real-time benchmark (Medium, 2026) shows CLI mode adds ~600ms of model-load per call, while warm-server mode lands at ~100ms mean / P95 RTF 0.024 for small on Windows — the best latency/accuracy/tail balance of any engine he tested.

Option C — dedicated whisper server projects. Several community HTTP wrappers exist (e.g. ahmetoner/whisper-asr-webservice, fedirz/faster-whisper-server with an OpenAI-compatible /v1/audio/transcriptions endpoint). These let your chat backend treat STT as a black box and swap implementations behind a stable API.

Takeaway. For a Python chat backend on NVIDIA: faster-whisper with large-v3-turbo INT8 (~2 GB VRAM, ~10× realtime, ~2.8% WER) is the sweet spot. Add WhisperX only if you need diarization/timestamps. On Apple Silicon: whisper.cpp with Metal+CoreML is the only first-class option (faster-whisper has no Metal backend and falls back to CPU).

2. Other modern local STT (2025–2026 SOTA)

The HuggingFace Open ASR Leaderboard is the authoritative ranking. As of mid-2025, NVIDIA Parakeet-TDT-0.6B-v2 held #1 with 6.05% avg WER.

2.1 NVIDIA Parakeet (NeMo) — current SOTA for English

  • Parakeet-TDT-0.6B-v2 — 600M params, FastConformer-TDT decoder. #1 on HF Open ASR Leaderboard (May 2025) with 6.05% avg WER (greedy, no external LM): AMI 11.16, Earnings-22 11.15, GigaSpeech 9.74, LibriSpeech clean 1.69 / other 3.19, SPGI 2.17, TED-LIUM 3.38, VoxPopuli 5.95. RTFx 3,386 (~56 min of audio/sec at batch 128). License CC-BY-4.0 (commercial OK). Source: HF model card + NVIDIA dev blog Jun 2025.
  • Parakeet-RNNT-1.1B Multilingual — 25 languages, built-in punctuation, supports streaming. Riva NIM container needs >50 GB VRAM for all modes; you can export just ofl (offline, ~25.6 GB) + str-thr (streaming-throughput, ~14.8 GB) for memory-constrained GPUs. Source: NVIDIA NIM docs.
  • Parakeet-unified-en-0.6b — single RNN-T model that does both offline and streaming with latency as low as 160ms. Offline WER 5.91%; at 240ms latency 6.92%, at 160ms 8.44%. Source: HF model card. Best single-model streaming option if you commit to NeMo.

Local serving is via the NeMo toolkit (Python, pip install nemo_toolkit['asr']): 

import nemo.collections.asr as nemo_asr
asr_model = nemo_asr.models.ASRModel.from_pretrained("nvidia/parakeet-tdt-0.6b-v2")
asr_model = asr_model.to("cuda").to(torch.bfloat16)
transcript = asr_model.transcribe(["audio.wav"])
Source: Towards AI walkthrough (May 2025) — transcribes a 2:37 clip in <2s on an RTX 3050 laptop. Note: Parakeet is GPU-only (CUDA); CPU inference is impractical. Not deployable via Ollama — it's a NeMo/PyTorch model, not a GGUF chat model.

2.2 Moonshine (Useful Sensors) — best for low-latency / edge / streaming

usefulsensors/moonshine (now rebranded Moonshine Voice, repo moonshine-ai/moonshine). Built from scratch for live voice interfaces, not a Whisper port.

  • Variable-length input — no Whisper 30-second zero-padding tax; processes only the audio you give it. 5× faster than Whisper on 10-second clips per the launch post (Pete Warden, Oct 2024).
  • Streaming with KV-cache — gen-2 models cache encoder + decoder state across incremental audio chunks, dropping latency further.
  • Better than Whisper Large v3 at a fraction of the size. Per the repo README: their English Medium Streaming model (250M params) beats Whisper Large v3 (1.5B) on the HF Open ASR Leaderboard. Tiny models go down to ~1 MB for constrained devices.
  • Cross-platform C++ core on ONNX Runtime → Linux/macOS/Windows/iOS/Android, with Python/Swift/Java/C++ bindings. Best "one library for everything voice" option — bundles STT, intent recognition, TTS, diarization.
  • Also in HuggingFace Transformers as MoonshineForConditionalGeneration (docs) — uses RoPE instead of absolute position embeddings so it handles variable lengths without padding.

2.3 Voxtral (Mistral) — audio-LLM, not pure STT

Released July 2025, Apache 2.0. Launch post. Two sizes: Voxtral Small 24B (production) and Voxtral Mini 3B (edge/local). Runs locally via transformers or vLLM. It's an audio-understanding LLM, not a streaming ASR front-end — see §5.

3. Streaming / real-time STT

"Streaming" has two meanings; both matter for voice chat:

  1. Chunked partial results — transcribe 0.5–2s windows, emit evolving text as the user speaks. All Whisper runtimes support this naively (you feed chunks; you get a transcript per chunk), but the 30-second fixed window makes Whisper wasteful on short utterances.
  2. True streaming decoder — RNN-T / TDT architectures that emit tokens as audio flows, with no chunk boundary. This is what Parakeet-unified, Moonshine streaming, and NVIDIA's streaming NIMs provide.

3.2 Production guidance for a voice chat loop

  • Don't run whisper.cpp as a CLI per utterance. Model-load dominates latency. Run it as a persistent server. (Kuo, 2026)
  • VAD first, then transcribe. Silero-VAD (used by faster-whisper v1.2.1 and WhisperX) drops fewer words on noisy audio than whisper.cpp's energy-based VAD — measured ~0.3–1% WER difference on accented/noisy input. (TildAlice benchmark, 2026)
  • For <250ms end-of-utterance latency you need an RNN-T/TDT streaming model (Parakeet-unified, nemotron-streaming) or Moonshine's streaming arch. Chunked Whisper bottoms out around ~1s because each chunk needs a full forward pass.
  • For barge-in (user interrupts the assistant), you need partial results mid-utterance — only the true streaming decoders do this cleanly.

4. Gemma 3 "native audio" in Ollama — reality check

Important correction. "Gemma 3 with native audio support via Ollama" is not accurate as stated. The Gemma 3 family released in 2025 (4B/12B/27B) is text + vision (image) only — it has no audio encoder. Ollama's audio capability landed with Gemma 3n (the edge variant, convertible via convert/convert_gemma3n.go) and the newer Gemma 4 E2B/E4B models. The rest of this section uses the audio-capable checkpoints Ollama actually ships.

4.1 What Ollama actually supports for audio

Ollama added an audio capability to its model registry. From ollama/ollama — types/model/capability.go (commit 573386c):

const (
    CapabilityCompletion = Capability("completion")
    CapabilityTools      = Capability("tools")
    CapabilityInsert     = Capability("insert")
    CapabilityVision     = Capability("vision")
    CapabilityEmbedding  = Capability("embedding")
    CapabilityThinking   = Capability("thinking")
    CapabilityImage      = Capability("image")
    CapabilityAudio      = Capability("audio")   // ← audio modality
)

The integration test file integration/audio_test.go declares the audio-capable models Ollama tests against:

var defaultAudioModels = []string{
    "nemotron3:33b",
    "gemma4:e2b",
    "gemma4:e4b",
}

So in the current Ollama registry, the audio-capable chat models are gemma4:e2b, gemma4:e4b, and nemotron3:33b — not gemma3. (A Gemma 3n converter exists at convert/convert_gemma3n.go, so a community GGUF of Gemma 3n may also surface, but the official tags above are what's tested.)

4.2 How it works (API surface)

Ollama exposes audio two ways, both shown in the integration test:

(a) Native Ollama API — audio is passed in the images field of a chat message (audio bytes are encoded the same way as image bytes): 

req := api.ChatRequest{
    Model: model,
    Messages: []api.Message{
        {Role: "system", Content: "Transcribe the audio exactly as spoken..."},
        {Role: "user", Content: "What exact words are spoken in this audio?",
         Images: []api.ImageData{audio}},   // ← audio in the images array
    },
    Options: map[string]any{"temperature": 0, "seed": 123, "num_predict": 50},
}
Source: audio_test.go TestAudioTranscription.

(b) OpenAI-compatible endpoints — Ollama implements both /v1/audio/transcriptions (multipart file upload, Whisper-style) and /v1/chat/completions with input_audio content blocks (GPT-4o-audio-style): 

// /v1/chat/completions with input_audio
"messages": [{
    "role": "user",
    "content": [
        {"type": "input_audio",
         "input_audio": {"data": "<base64-wav>", "format": "wav"}}
    ]
}]
Source: TestOpenAIAudioTranscription & TestOpenAIChatWithAudio.

4.3 Quality and capability — what the tests tell us

The tests only assert that the model returns some expected keywords, not faithful transcription:

  • TestAudioTranscription — sends a clip saying "Why is the sky blue?" with a system prompt "Transcribe the audio exactly as spoken. Output only the spoken words. Do not answer any question in the audio." and only checks the response contains "sky" and "blue".
  • TestAudioResponse — same clip, no transcription instruction; checks the answer mentions scatter / light / blue / atmosphere / wavelength / rayleigh.

This is a weak bar. There are no published WER numbers for Gemma 4 E2B/E4B or Nemotron3 audio transcription, and no head-to-head against Whisper/Parakeet. Anecdotal reports (and the Voxtral-demo video linked in §5) put Gemma-class audio-LLM transcription "favorable but not Whisper-large quality." Treat audio-LLM transcription as conversational-grade, not ASR-grade.

4.4 Can it do transcription AND reasoning AND tool calling at once?

Ollama's Capability("tools") and Capability("audio") are orthogonal flags — nothing in the capability model prevents a model from advertising both. However:

  • The audio integration tests do not exercise tool calling alongside audio. The tool-calling tests (see docs/capabilities/tool-calling.mdx) use text-only models like qwen3.
  • Whether a specific audio-capable GGUF (gemma4:e2b/e4b, nemotron3:33b) actually emits well-formed tool-call tokens while receiving audio is model-dependent and unverified by Ollama's CI. You must test your chosen checkpoint empirically.
  • The Voxtral demos (§5) explicitly show function-calling from voice working in vLLM; Gemma 4 / Nemotron3 have no equivalent public demo.

4.5 Known limitations

  • No streaming partials. Audio-LLMs decode a full response token-by-token after consuming the whole clip; they don't emit partial transcripts mid-utterance.
  • Latency. Running a multi-billion-param LLM forward on every utterance is much slower than a dedicated ASR front-end. faster-whisper-turbo does a 30s clip in <1s; an audio-LLM takes seconds-to-tens-of-seconds depending on size.
  • VRAM contention. If your chat LLM (Gemma 4 12B QAT, per your models/ dir) and an audio-LLM coexist, you're paying for both. A separate small STT model is usually cheaper than a big audio-LLM.
  • Transcription faithfulness. Audio-LLMs are trained to understand audio, not to produce verbatim transcripts. They paraphrase, drop disfluencies, and sometimes answer instead of transcribe (the Ollama test uses a system prompt to suppress this).

5. Audio-LLMs with tool calling (the local question)

This is the crux: can a single local model receive audio AND emit tool calls, so the user can speak a query and the model both understands it and triggers a knowledge-graph retrieval tool?

5.1 Voxtral (Mistral) — YES, explicitly supported

From the launch announcement and the Voxtral-Small-24B model card:

Function-calling straight from voice: Enables direct triggering of backend functions, workflows, or API calls based on spoken user intents, turning voice interactions into actionable system commands without intermediate parsing steps.

The HF model card shows a working code example — Voxtral-Small-24B-2507 "Function Calling" section. The model is given a get_current_weather tool and an audio clip of someone asking about the weather in Madrid; the response contains a well-formed tool_calls object:

tools = [tool.to_openai()]
audio_chunk = file_to_chunk(weather_like)   # fn_calling.wav
user_msg = UserMessage(content=[audio_chunk]).to_openai()
response = client.chat.completions.create(
    model=model, messages=[user_msg],
    temperature=0.2, top_p=0.95,
    tools=[tool.to_openai()],
)
print(response.choices[0].message.tool_calls)  # → get_current_weather(location="Madrid", ...)

Local serving confirmed via transformers (VoxtralForConditionalGeneration) and vLLM. The Voxtral Mini 3B local testing video (Bijan Bowen, Jul 2025) demonstrates voice-driven Blackjack and reflex games using function calling, running locally on a 5090 laptop GPU via vLLM (~20 GB VRAM with the 32k context pre-allocated; ~9 GB minimum per the model card). Real-world local Voxtral-Mini-3B deployments in the wild (e.g. matthiasn/lotti services/voxtral-local, rishikanthc/Scriberr voxtral adapter, blakkd/faster-whisper-hotkey with int8/int4 quantization) confirm it runs locally on consumer hardware.

Caveats:

  • "Experimental function calling support" per the 24B card. System prompts not yet supported in chat mode.
  • Voxtral is not in the Ollama registry as of this writing — you serve it via vLLM or transformers. (llama.cpp maps Qwen2AudioForConditionalGeneration to the ultravox arch per conversion/__init__.py, so GGUF paths exist for Qwen2-Audio, but Voxtral support is less mature.)
  • Context: 32k tokens → up to 30 min audio for transcription, 40 min for understanding.

5.2 Qwen2-Audio — audio + chat, tool-calling not first-class

Qwen/Qwen2-Audio-7B-Instruct — 7B, Apache 2.0, audio + text chat. Widely deployed locally (see xinference, intel/ipex-llm with INT4, vllm-compressor quantization example, sglang). Runs at ~4-bit on a single consumer GPU.

Tool calling: Qwen2-Audio is built on the Qwen2-7B chat backbone, which does support tool calls in text mode — but the official Qwen2-Audio model card and examples do not demonstrate tool calling interleaved with audio input. There's no equivalent of Voxtral's "speak → tool_call" example. You'd be relying on the underlying Qwen2 LM head emitting tool tokens while audio embeddings are in context; unverified and likely brittle. For a production voice+tools system, Voxtral is the safer bet.

5.3 Qwen2.5-Omni — the omni-modal local option

QwenLM/Qwen2.5-Omni — 7B (and 3B), Apache 2.0. Audio + vision + text in, text + speech out. Ranked #1 open-source on MMAU (audio understanding/reasoning) in June 2025. Served via vLLM (vllm serve Qwen/Qwen2.5-Omni-7B --omni) or MNN for edge. GPTQ-Int4 / AWQ 4-bit variants cut VRAM >50% (per the README). Tool calling: built on Qwen2.5, which has strong text tool-calling — but like Qwen2-Audio, the public omni examples focus on chat, not voice→tool. Plausible but unverified for voice→tool_call; test empirically.

6. Architecture patterns: voice in + knowledge graph

The core problem: the user speaks → the system must retrieve from a knowledge graph → answer. Three patterns exist in the wild.

Pattern A — Transcribe-then-Retrieve (the dominant pattern)

audio → [STT model] → text query
                         ↓
                   [LLM with tools]
                         ↓
              tool_call: query_knowledge_graph(text)
                         ↓
                   [graph retrieval]
                         ↓
                   LLM generates answer
                         ↓
                   [TTS] → audio out
  • STT and LLM are decoupled. STT picks the best-of-breed ASR (faster-whisper-turbo, Parakeet, Moonshine); the LLM is your existing chat model (Gemma 4 12B QAT, per your models/ dir).
  • Tool calling is trivial because the LLM only ever sees text — you already have a working tool-calling setup.
  • Latency budget: STT (~0.3–1s) + 1 LLM turn for tool-call (~0.5s) + graph retrieval (~50–200ms) + 1 LLM turn for answer (~1–3s) + TTS (~0.5s). Total ~2.5–5s. Acceptable for turn-based voice chat; too slow for true barge-in.
  • This is what most production voice-RAG apps do, including the Llama-Stack voice demos and most "voice agent" templates.

Pattern B — Audio-native LLM with tools (Voxtral-style)

audio → [Voxtral] → tool_call(query_knowledge_graph, args=...)
                          ↓
                    [graph retrieval]
                          ↓
                    Voxtral generates answer (text or audio-out via TTS)
  • One fewer hop — no separate STT step; the audio-LLM emits the tool call directly from the audio embedding.
  • Quality tradeoff: Voxtral's transcription is good but not Parakeet/Whisper-large good. For domain vocabulary (entity names in your knowledge graph), a dedicated ASR with custom vocabulary biasing (Voxtral API's context_bias, or Whisper with prompt prefix) will outperform an audio-LLM that has to both transcribe AND decide to call a tool.
  • Latency: one model forward instead of two, but that forward is over a multi-B-param audio-LLM, so wall-clock is often longer than Pattern A's two small forward passes.
  • Best when the query is conversational and the graph entities are common words. Worst when entity names are rare (proper nouns, IDs) — the audio-LLM may hallucinate them and pass garbage to the tool.

Pattern C — Hybrid: ASR for transcript + audio-LLM for intent (advanced)

audio → [fast ASR] → transcript (verbatim, for graph query)
       → [audio-LLM] → intent / tool selection (uses prosody, emotion)
                          ↓
              graph retrieval using transcript
                          ↓
              text LLM generates answer
  • Use ASR for the faithful string the graph needs (entity matching is string-based).
  • Use the audio-LLM only to classify intent / detect emotion / decide whether to call a tool — leveraging the audio signal ASR throws away.
  • Overkill for most apps; only worth it if you need prosody/emotion-aware routing.

Pattern D — Audio-native RAG (research, not production)

True "audio-native RAG" — embedding audio queries directly against an audio-indexed corpus, never going through text — exists in research (audio retrieval via CLAP-style audio embeddings, e.g. Audio Flamingo / AF-CLAP mentioned in this 2026 overview) but is not mature for knowledge-graph Q&A. Knowledge graphs are text/structured-data artifacts; trying to query them with audio embeddings buys nothing and loses entity precision. Skip for now.

Recommendation for your app. Use Pattern A (transcribe-then-retrieve) as the backbone — it's the only pattern that lets you use a SOTA ASR and your existing tool-calling LLM and your existing knowledge graph unchanged. Treat audio-native tool calling (Pattern B / Voxtral) as a future experiment once the basic voice loop works.

7. Recommendation

Recommended architecture

STT front-end: faster-whisper with large-v3-turbo, INT8, served in-process from your FastAPI api/services/ (replacing the current llama-based transcribe endpoint). On your hardware this is ~2 GB VRAM, ~10× realtime, ~2.8% WER. If you want to push quality further and have CUDA headroom, run Parakeet-TDT-0.6B-v2 via NeMo in a sibling service (6.05% leaderboard WER, CC-BY-4.0, but GPU-only and a heavier dependency).

Chat LLM: keep your existing Gemma 4 12B QAT (in models/gemma4-12b-qat) served by Ollama, with its existing tool-calling setup for knowledge-graph retrieval. Do not route audio through the chat LLM — it's slower, less accurate at transcription, and untested for tool-calling-while-audio by Ollama's own CI.

Why not Gemma 4 E2B/E4B audio: it adds a second multi-B-param model to VRAM, gives no published WER, no streaming partials, and no demonstrated voice→tool_call. A dedicated ASR + your existing tool-calling text LLM is strictly better on every axis that matters for a knowledge-graph app.

Future experiment (optional, low-risk): stand up Voxtral Mini 3B via vLLM in a separate service as a Pattern-B alternative. It's the only local audio-LLM with documented voice→tool_call.
