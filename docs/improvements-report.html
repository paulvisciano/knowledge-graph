Key insight from live image-upload test (5 images, 2026-07-20): The bottleneck is not EXIF extraction (fast) — it's the VLM AI analysis step, serialized by VLM_MAX_CONCURRENT=1 (api/services/processor.py:105). Images appear stuck at extracting_exif in the UI but are actually queued behind the single VLM slot. The UI stage label reflects the last yielded SSE event, not the real blocking step — a backend/UI stage mismatch.
  • EXIF is cheap → run for all images immediately in parallel (already does, ~fast)
  • AI analysis is expensive (~88s/call) → the real serialization point. Increase VLM_MAX_CONCURRENT, batch, or queue with clear "waiting for AI" stage
  • Images on disk → render in graph immediately via setPhotoImage(nodeId, dataUrl) (line 261) — already works, but uses heavy base64 dataUrl; should use the lighter thumbnailUrl blob URL
  • No attachment previews in the chat UI — attachments state has no render loop; AttachmentPreview.svelte exists but is never imported
▶ Performance recommendations (16GB RAM constraint) 10 items · 4 tiers

Constraint: 16 GB total system RAM. Observed baseline: host VLM (Gemma-4-12B Q4_K_M via llama-server at :8080) holds ~7–8 GB; all 5 containers together ~1 GB; macOS + browser ~2 GB. Total ~11.5–12 GB resident, ~3–4 GB headroom. The VLM is both the bottleneck (~88s/call, serialized by asyncio.Semaphore(1)) and the memory hog — most generic "raise concurrency" advice would OOM. Recommendations below are ordered by ROI under this ceiling.

Tier 1 — Pure wins faster + less RAM, no tradeoff

1Drop base64 dataUrls in the graph; use the thumbnailUrl blob: URL already created at file-utils.ts:51. 5 photos = ~40 MB of duplicated base64 strings across graphStore + imageProcessingStore + DOM. Blob URLs are ~0 KB (file-backed). Frees RAM and speeds first paint. Refs: finding #2 · ui/src/routes/+page.svelte:261 setPhotoImage(nodeId, att.dataUrl)
2Backoff + single-call poll in wait_for_lighrag_processing. Halves LightRAG load, stops log spam. Exponential 3→15s with jitter. Bounded at 300s. Refs: finding #6 · api/services/processor.py:1562–1604
3Runtime stuck-job watchdog. A hung VLM call currently ties a semaphore slot forever. A 10-min watchdog that marks stuck jobs failed and releases the slot recovers throughput without spending more memory. Refs: finding #7 · api/services/job_manager.py:259–285 (_STUCK_JOB_TIMEOUT_SECONDS=30min, startup-only)

Tier 2 — Same RAM, better perceived perf UX wins, memory-neutral

4Split EXIF from AI; emit a queued_for_ai stage. EXIF is fast and cheap — run it for all images upfront in parallel (asyncio.gather, no semaphore needed; just Pillow + tag reads), then queue for the VLM. The UI immediately shows all 5 photos in the graph with real EXIF labels; the AI step fills descriptions as the single VLM slot frees. Biggest perceived-perf win in the report, ~0 extra RAM. Refs: findings #3, #4 · api/services/processor.py:871+ (sequential EXIF→faces→VLM) · image-processing.svelte.ts:135–148 mapEventToStage
5Stream the VLM response (SSE text deltas) instead of buffering 1024 tokens. llama-server supports stream: true. Show partial description in the UI as it generates — first token in ~1–2s instead of 88s. Same wall-clock total, dramatically better UX. Costs nothing in RAM (connection is already held open). Refs: api/services/processor.py:923–1010 describe_image_with_vlm payload — add "stream": true

Tier 3 — Real throughput gains requires a model/server choice

6Smaller model → more slots. Drop Gemma-4-12B to a 4B/3B Q4 (~2.5 GB resident). ~5 GB headroom → llama-server --parallel 2 (two slots in one process, ~3 GB total). Two VLM calls in flight: ~88s each but 2× throughput → 5 images in ~3.5 min instead of ~7 min. Quality drops on descriptions; EXIF+labels carry most of the graph value. Refs: finding #41 · api/services/processor.py:105–106 _vlm_semaphore = asyncio.Semaphore(1)
7Continuous batching via a dedicated server (vLLM / llama.cpp --cont-batching). One 12B model, 2–3 concurrent requests share ~8 GB + small per-request KV cache. Same RAM as today, true parallelism instead of serialized semaphore. The durable long-term answer — non-trivial infra swap. Refs: finding #41
8Batch multiple images into one VLM call. Gemma-4 supports multi-image input. Send 3–5 images in one chat completion ("describe each image N"). One forward pass, one ~88s round-trip covers all 5 instead of 5×88s. RAM-neutral. Caveat: per-image quality drops and you lose per-image retry isolation. Best as the initial pass; fall back to single-image calls for failures. Lowest-risk experiment — one prompt change, measurable in an hour. Refs: api/services/processor.py:943–976 payload construction

Tier 4 — Don't bother under 16 GB avoid — OOM risk or zero gain

9Raising VLM_MAX_CONCURRENT alone. With one llama-server slot, a second caller just blocks inside the HTTP slot. Zero throughput gain, slight memory overhead from the queued connection. Don't. Refs: findings #3, #41
10Second VLM container. Two 12B instances = ~14 GB just for models — OOM. Don't. Refs: docker-compose.yml host VLM at :8080

Recommended sequence: Tier 1 + #4 + #5 first (blob URLs, backoff poll, watchdog, parallel EXIF + immediate graph render, streaming VLM) — none touch the 16 GB ceiling and together they change the felt experience from "5 photos stuck on 'Extracting EXIF…' for 7 min" to "all 5 photos appear in 3s with labels, descriptions stream in over ~7 min, UI never lies about what's happening." Then, to cut wall-clock, pick one of Tier 3 — lean #8 (batched-prompt) as the lowest-risk experiment; if quality holds, done; if not, escalate to #7 (vLLM cont-batching) as the durable fix.
