Knowledge Graph

Personal knowledge graph stack: LightRAG + local LLMs + real-time image processing + 3D graph UI

Set up the full stack on a new machine. Prerequisites: macOS with Apple Silicon, ~10 GB free disk space for model files, and the following tools:

ToolInstallWhy
Homebrew /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)" Package manager — needed for llama.cpp
Docker Desktop brew install --cask docker Runs LightRAG, PostgreSQL, API, Nexus UI in containers
llama.cpp brew install llama.cpp Runs LLM, embedding, and reranker models on Metal GPU

Docker Desktop post-install: After installing, open Docker Desktop and wait for the engine to start (whale icon in menu bar stops animating). The stack auto-configures the VM memory from DOCKER_VM_MEMORY_MIB in .env (default 1.5 GiB) via scripts/setup-docker.sh — no manual Settings → Resources step needed.

1. Clone & Configure

# Clone with submodules (includes patched LightRAG fork)
git clone --recurse-submodules https://github.com/paulvisciano/knowledge-graph.git
cd knowledge-graph

# If you already cloned without --recurse-submodules:
git submodule update --init --recursive

# Copy the environment template and customize as needed
cp .env.example .env

All config lives in a single .env file — model names, ports, storage backends, everything. No separate LightRAG config needed.

2. Download Model Files

You need four GGUF model files (~10 GB total). Place them under ~/models/ (or set MODEL_DIR in .env):

ModelFileSizeHuggingFace Search
LLM (Gemma 4 12B)Gemma-4-12B-OBLITERATED-Q4_K_M.gguf~6.9 GBSearch "Gemma-4-12B OBLITERATED GGUF" on HuggingFace
Embedding (BGE-M3)bge-m3-Q4_K_M.gguf~1.1 GBSearch "bge-m3 GGUF" on HuggingFace (e.g. gpustack/bge-m3-GGUF, doof-ferb/bge-m3-gguf)
Reranker (bge-reranker-v2-m3)bge-reranker-v2-m3-Q4_K_M.gguf~418 MBSearch "bge-reranker-v2-m3 GGUF" on HuggingFace
Vision projection (mmproj)mmproj-BF16.gguf~2.3 GBSearch "Gemma 4 12B mmproj GGUF" on HuggingFace

Why mmproj? Gemma 4 has a vision architecture, but in llama.cpp the vision encoder weights are a separate file. The base GGUF contains only the language model — mmproj-BF16.gguf is the vision projection that maps image pixels into the model's embedding space. Without it, llama-server cannot process images. The start script passes it via --mmproj.

⚠️ mmproj compatibility: The Gemma 4 12B mmproj uses the gemma4uv projector type, which requires llama.cpp v9750+ (June 2026 or later). Older versions crash with unknown projector type: gemma4uv. If you see this error, upgrade: brew upgrade llama.cpp. If you can't upgrade, move the mmproj aside (mv mmproj-BF16.gguf mmproj-BF16.gguf.bak) — the start script will skip --mmproj and the API will fall back to EXIF-only image processing (location, date, camera metadata still works, but no visual descriptions).

3. Install llama.cpp (Host)

# Install llama.cpp via Homebrew (provides llama-server)
brew install llama.cpp

# Verify — must be v9750+ for Gemma 4 12B vision support
llama-server --version

# If older than v9750, upgrade:
brew upgrade llama.cpp

⚠️ Version requirement: llama.cpp v9750+ (June 2026 or later) is required for the gemma4uv projector type used by Gemma 4 12B's mmproj. Older versions crash with unknown projector type: gemma4uv. If you can't upgrade, move the mmproj aside — the stack will run in text-only mode with EXIF fallback for images.

4. Start the Full Stack

The entire stack — Docker containers and host llama-servers — starts with a single command. The script first ensures Docker Desktop VM memory matches .env, then builds and starts the containers, then launches the llama-servers in the foreground for Metal GPU acceleration.

# Start everything: Docker containers + host llama-servers
./scripts/start-all.sh

# Containers: LightRAG, PostgreSQL, API, Nexus UI, MCP
# Host: LLM (:8080), embeddings (:8081), reranker (:8082), whisper (:8090)

# Press Ctrl+C to stop the llama-servers
# Run ./scripts/stop-all.sh to tear down everything (containers + servers)

First build takes a few minutes for the LightRAG image (Python + Rust deps). First model load takes ~30s. The llama-servers run in the foreground so Ctrl+C stops them cleanly — use stop-all.sh to also tear down the Docker containers.

Why PostgreSQL?

Persistent, queryable storage for vectors, KV, and doc status. NetworkX for graph storage (AGE extension needed for PG graph — not worth the custom image). HNSW index for fast vector search.

Why host llama-servers?

Apple Silicon Metal GPU acceleration can't run in Docker. llama-server runs natively on the host with full Metal support. LightRAG in Docker connects via host.docker.internal.

Why local build?

docker-compose.yml builds from ./lightrag source rather than pulling ghcr.io/hkuds/lightrag:latest. This lets us modify LightRAG code and rebuild with docker compose up -d --build.

Why VLM-based image ingestion?

LightRAG's /documents/upload only accepts text-based file types (txt, pdf, md, etc.). Image files (.jpg, .png) require the MinerU or Docling parser engines, neither of which is configured in our stack. Instead of deploying an external parser, we send images directly to the VLM (Gemma 4 with mmproj on the same llama-server) to generate a description, then insert that description as text into LightRAG via /documents/text. EXIF metadata (location, date, camera) is injected as separate entity nodes linked to a central Photo node.

Why a combined document instead of separate VLM + metadata docs?

Initially, VLM descriptions and EXIF metadata were inserted as two separate LightRAG documents. This created disconnected subgraphs — "Saint Petersburg" from metadata had no edge to "Backyard" from the VLM description, because LightRAG's entity extraction LLM treated them as unrelated. Combining them into a single document (with EXIF context prepended to the VLM prompt) and injecting EXIF entities as graph nodes with edges to a Photo hub node ensures everything is connected.

Why EXIF entity injection + visual entity linking?

Even in a combined document, LightRAG's entity extraction can't reliably infer that "Saint Petersburg" (from EXIF) and "Backyard" (from VLM) are about the same photo. The reference implementation (memory-search-latest) solves this by programmatically creating Date, Location, and Camera entity nodes with edges (taken_on, taken_at, taken_with) to the photo node, and linking visual entities (Backyard, Person, etc.) back to the photo via appears_in edges. This makes the Photo node the central hub connecting all subgraphs.

Why patch LightRAG's openai.py?

The OpenAI Python SDK (v2.36.0) returns a list instead of CreateEmbeddingResponse when the base_url doesn't end with /v1. Since our EMBEDDING_BINDING_HOST=http://host.docker.internal:8081 lacks the /v1 suffix, every embedding call crashed with 'list' object has no attribute 'data'. The fix normalizes both response shapes and handles the extra nesting in the list case. This is a known SDK compatibility issue with OpenAI-compatible servers.

Why standalone processors?

Image processing modules live in processors/ outside the lightrag/ submodule. They talk to LightRAG via HTTP API, not by patching its source. This means LightRAG updates don't break image processing.

Why Python 3.11 venv?

macOS ships Python 3.9 which has a TensorFlow mutex crash. The project uses a .venv/ with Python 3.11 (installed via Homebrew) where TensorFlow 2.21 + DeepFace work correctly. start-api.sh auto-detects the venv.

Why a persistent job queue?

Image processing involves multiple stages (EXIF extraction → entity creation → VLM description → face recognition) that can take 30+ seconds. A SQLite-backed job queue ensures processing state survives server restarts, prevents duplicate processing, and provides a clean API for tracking progress via SSE. Jobs are created on upload and their status and events are persisted, so clients can reconnect and resume monitoring.

Why nginx reverse proxy?

In production, nginx serves the static SvelteKit build and proxies all API calls — LightRAG, KG API, llama-server, MCP, and WebSocket sync — through a single port (:5180). No CORS issues, no mixed content, clean routing. In dev, Vite's proxy handles the same routes.

End-to-End Workflow: Audio Message → Response (with BGE-M3)

Describes both the current audio path (traced from source) and the proposed path that routes voice messages through the knowledge graph with BGE-M3. Today, audio bypasses the RAG pipeline entirely — Gemma answers from its own knowledge. The proposed path wires audio into LightRAG so the embedding model gates retrieval.

Current Path — What Actually Happens Today

Traced through ui/src/routes/+page.svelte, ui/src/lib/utils/audio-recording.ts, docker/nginx/nginx.conf. Audio does not go through LightRAG, the embedding model, vector search, or the reranker.

2. Encode  PCM chunks → audioBufferToWav() → WAV blob → blobToBase64() → base64 data URL

3. Send to Gemma handleSend(audioUrl, audioData, 'wav') → streamAssistantResponse()
            → builds { type: 'input_audio', input_audio: { data, format: 'wav' } } content part
            → POST /v1/chat/completions → nginx → llama-server :8080 (Gemma 4 12B)
            +page.svelte line 1053: if ($chatMode === 'kg-direct' && !audioData) — audioData forces streamAssistantResponse,
            never streamKGChatResponse. No transcription step — Gemma handles the audio modality natively.
            Note: transcribeAudio() and the /v1/audio/transcriptions endpoint constant exist in the codebase
            but are dead code — never called. No Whisper model is loaded by start-llama-servers.sh.

4. Generate Gemma processes audio + system prompt + conversation history → streams text response tokens
            answers from Gemma's own knowledge — no knowledge-graph context, no retrieved chunks

5. Render streamed tokens → chat panel via requestAnimationFrame throttled flush
            → AudioPlayer.svelte shows the original voice message alongside the response

Gap: voice messages never query the knowledge graph. A user asking "show me photos of Paul at the beach" by voice gets a generic Gemma answer, not a graph-grounded one. The embedding model, vector search, and reranker are not involved.

Proposed Path — Audio Routed Through the Knowledge Graph (with BGE-M3)

Requires two code changes: (a) Gemma transcribes the audio to text, (b) the transcript routes through streamKGChatResponse() (LightRAG query) instead of streamAssistantResponse() only. Steps 6-8 are where BGE-M3 replaces nomic-embed-v2.

1. Capture  User taps mic in Nexus UI → AudioRecorder (AudioWorklet) captures PCM samples
            ui/src/lib/utils/audio-recording.ts — PCMProcessor worklet → Float32Array chunks

2. Encode  PCM chunks → audioBufferToWav() → WAV blob → base64 data URL
            persisted as audioData in the message store via sync-client (durable across reloads)

3. Transcribe WAV → { type: 'input_audio', input_audio: { data, format: 'wav' } } → POST /v1/chat/completions
            → Gemma 4 12B on :8080 handles audio natively → returns transcript text (may be any language)
            no separate Whisper model — Gemma's audio modality produces the transcript directly

4. Query LightRAG transcript → streamKGChatResponse() → Nexus /api/lightrag/query/stream → LightRAG :9621
            Nexus proxies through nginx to LightRAG's streaming query endpoint

5. Extract keywords LightRAG sends transcript to Gemma (:8080) with a keyword-extraction prompt
            → returns high-level + low-level keywords
            operate.py → extract_keywords_only()

6. Embed query LightRAG calls EMBEDDING_BINDING_HOST (:8081) → BGE-M3 embeds the transcript + keywords
            → 1024-dim dense vectors (batched: query + ll_keywords + hl_keywords in one call)
            ★ BGE-M3 benefit cross-lingual: a Russian/Spanish transcript embeds in the same
            space as the multilingual VLM descriptions and EXIF place names — no English-only bottleneck

7. Vector search 1024-dim query vectors → pgvector HNSW over PostgreSQL
            → top-k candidate entities, relations, and chunks (combined photo docs, EXIF nodes, VLM descriptions)
            ★ BGE-M3 benefit 1024 dims → finer neighbor ranking; sparse/ColBERT functions
            preserve exact proper nouns ("Paul", "Nikon Z6", "Backyard") that 768-dim dense-only blurs

8. Rerank candidates → bge-reranker-v2-m3 on :8082 (Jina /v1/rerank)
            → re-scored and reordered
            ★ BGE-M3 benefit same M3 family as the embedder → tokenizer and
            representation space agree, so the reranker refines an ordering the embedder already
            produced instead of re-tokenizing against a mismatched vocabulary

9. Generate top reranked chunks + conversation history → Gemma 4 12B on :8080
            → streamed answer tokens grounded in retrieved knowledge-graph context
            better-grounded context (steps 6-8) → fewer hallucinated entities, correct photo/person/place references

10. Render answer stream → Nexus chat panel; AudioPlayer.svelte shows the original voice message
            SSE/streaming response proxied back through nginx :5180

Net Effect on the Audio → Response Path

Today: audio messages bypass the knowledge graph entirely — Gemma answers from its own knowledge with no retrieved context. The embedding model is not involved. Voice queries like "show me photos of Paul at the beach" get generic answers, not graph-grounded ones.

After the two changes (route audio through LightRAG + switch to BGE-M3): Gemma transcribes the audio natively (no Whisper needed), the transcript enters the LightRAG query pipeline, and BGE-M3 embeds it in a 100+-language 1024-dim space. Vector search returns language-consistent, exact-entity-matching candidates; the reranker (bge-reranker-v2-m3) refines them on a consistent M3 tokenization; and Gemma generates a graph-grounded answer. The UI, audio capture, and Gemma's audio modality are unchanged — the gains come from (a) routing audio into RAG at all, and (b) better retrieval quality at steps 6-8.

How the Embedding Model Fits Into the Query Pipeline

This table describes the LightRAG text-query pipeline — the path taken when a user types a query (or when audio is routed through LightRAG per the proposed path above). The LLM (Gemma 4 12B on :8080) and the embedding model (BGE-M3 on :8081) do different jobs. Gemma is called twice — once to parse the query into keywords, once to write the final answer. The embedding model is called once — but that single call decides what Gemma sees. The reranker (:8082) is called once to re-score candidates. Traced through the LightRAG source (lightrag/operate.py, lightrag/kg/postgres_impl.py, lightrag/utils.py):

StepModelPortSourceWhat happens
1. Keyword extraction Gemma (LLM) :8080 operate.py → extract_keywords_only() Sends the query to Gemma with a keyword-extraction prompt → returns high-level + low-level keywords (e.g. "photos of Paul at the beach" → hl: ["beach","summer"], ll: ["Paul"])
2. Batch embed BGE-M3 (embedding) :8081 operate.py → _perform_kg_search() Embeds up to 3 texts in ONE call: the raw query (→ query_embedding), ll_keywords (→ ll_embedding), hl_keywords (→ hl_embedding). Batched to avoid 2-3 sequential round-trips.
3. Vector search pgvector HNSW PostgreSQL postgres_impl.py → PGVectorStorage.query() Three separate HNSW cosine searches using those embeddings: entities VDB (by ll_embedding), relationships VDB (by hl_embedding), chunks VDB (by query_embedding, mix mode only). The embedding model's output IS the search key — SQL: ORDER BY content_vector <=> $4.
4. Graph traversal NetworkX — operate.py → _get_node_data() / _get_edge_data() From the top-k entity nodes found by vector search, fetches node metadata, degrees, and related edges from the NetworkX graph. NO embedding or LLM call — pure graph hops.
5. Rerank bge-reranker-v2-m3 :8082 utils.py → apply_rerank_if_enabled() Takes the merged candidate chunks + the original query and asks the reranker to re-score them. A SEPARATE model from the embedder — it scores query↔document relevance directly (cross-encoder).
6. Context build none — operate.py → _apply_token_truncation → _merge_all_chunks → _build_context_str Truncates to fit the LLM context window, builds a formatted context string with entities, relations, and chunks. NO model calls — pure data assembly.
7. Answer generation Gemma (LLM) :8080 operate.py → kg_query() Sends the system prompt (with assembled context) + user query to Gemma → streams the answer back.

The Key Takeaway

The embedding model is the retrieval gatekeeper. Gemma extracts keywords, but BGE-M3 decides which entities, relations, and chunks get retrieved from PostgreSQL. The LLM never sees the raw document store — it only sees what the embedding model's vector search surfaced (step 3). So if BGE-M3 returns poor matches (wrong language, blurred proper nouns), Gemma gets bad context and produces a bad answer regardless of how good the LLM is.

This is exactly why the BGE-M3 switch matters: steps 2-3 are where retrieval quality is decided, and that's purely the embedding model's job. Gemma runs identically before and after — but it receives better-ranked input. With BGE-M3, step 2 embeds in a 100+-language 1024-dim space (vs nomic's English-oriented 768-dim), and step 3's HNSW search returns more accurate, language-consistent candidates — so step 7's answer is grounded in better context with no LLM change.
