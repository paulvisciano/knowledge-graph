Knowledge Graph

Personal knowledge graph stack: LightRAG + local LLMs + custom UI

Current Status

Phase 1 — Complete Docker + LightRAG

Vanilla LightRAG stack running in Docker with host llama-servers on Metal GPU.

Roadmap

Phase 1 ✓ Docker + LightRAG

Get the core stack running: LightRAG in Docker, llama-servers on host (Metal GPU), file-based storage, basic document ingestion and querying. Validate the full pipeline end-to-end — insert, query, and verify the knowledge graph builds correctly.

Deliverable: Working LightRAG instance with local LLMs, accessible via API and built-in WebUI.

Phase 2 — Next Image Processing

Add support for image ingestion into the knowledge graph. Images get processed by the VLM (Gemma 4 12B) to extract descriptions, objects, scenes, and relationships — then stored as entities/edges in the KG alongside text documents.

Deliverable: Pipeline that accepts images, generates VLM descriptions, and inserts them into the KG.

Phase 3 — Planned Llama UI + Knowledge Graph Integration

Integrate the Llama UI with LightRAG so the chat interface can query and display the knowledge graph. Users can have natural-language conversations that pull from the KG, with citations and entity references surfaced inline.

Deliverable: Chat UI connected to LightRAG API — query the KG conversationally with context-aware responses.

Phase 4 Custom Chat + KG UI

Port or build from scratch a custom UI that combines chat and knowledge graph visualization. Think split-pane or tabbed layout: conversational interface on one side, interactive graph explorer on the other. Entities and relationships clickable and explorable.

Deliverable: Custom web UI at localhost:3000 with chat + KG exploration.

Phase 5 Enhanced Image Intelligence

Supercharge image processing with deep metadata extraction: EXIF data (camera, timestamps, GPS coordinates), face recognition and clustering, location detection and reverse geocoding, object detection with bounding boxes, OCR for text in images, and scene classification. All extracted as structured entities in the KG.

Deliverable: Rich image pipeline that turns photos into queryable knowledge — "find all photos from summer 2024 with faces" or "show me everything taken near Central Park."

Quick Start

# 1. Configure
cp .env.example .env

# 2. Start llama-servers on host
./scripts/start-llama-servers.sh

# 3. Start LightRAG
docker compose up -d

# 4. Open WebUI
open http://localhost:9621

# Stop everything
./scripts/stop-all.sh

Key Decisions

Why file-based storage?

Starting simple. NanoVectorDB + NetworkX + JsonKV require zero infra — no database to manage. The docker-compose.yml has a commented-out PostgreSQL service ready to uncomment when needed. Switching requires re-ingesting documents.

Why host llama-servers?

Apple Silicon Metal GPU acceleration can't run in Docker. llama-server runs natively on the host with full Metal support. LightRAG in Docker connects via host.docker.internal.

Why local build?

docker-compose.yml builds from ./lightrag source rather than pulling ghcr.io/hkuds/lightrag:latest. This lets us modify LightRAG code and rebuild with docker compose up -d --build.

Why nomic over bge-m3?

768-dim vs 1024-dim. Smaller, faster on Apple Silicon. With reranking enabled the quality gap is negligible for English content. bge-m3 is better for multilingual — switch requires re-ingestion.
