Knowledge Graph

Personal knowledge graph stack: LightRAG + local LLMs + real-time image processing + 3D graph UI

Getting Started

Set up the full stack on a new machine. Prerequisites: macOS with Apple Silicon, ~10 GB free disk space for model files, and the following tools:

ToolInstallWhy
Homebrew /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)" Package manager — needed for llama.cpp
Docker Desktop brew install --cask docker Runs LightRAG, PostgreSQL, API, Nexus UI in containers
llama.cpp brew install llama.cpp Runs LLM, embedding, and reranker models on Metal GPU

Docker Desktop post-install: After installing, open Docker Desktop and wait for the engine to start (whale icon in menu bar stops animating). Allocate at least 8 GB RAM in Settings → Resources. The API container uses DeepFace for face recognition which needs significant memory.

1. Clone & Configure

# Clone with submodules (includes patched LightRAG fork)
git clone --recurse-submodules https://github.com/paulvisciano/knowledge-graph.git
cd knowledge-graph

# If you already cloned without --recurse-submodules:
git submodule update --init --recursive

# Copy the environment template (edit paths/ports if needed)
cp .env.example .env

# Copy the LightRAG config template
cp lightrag/env.example lightrag/.env

3. Install llama.cpp (Host)

# Install llama.cpp via Homebrew (provides llama-server)
brew install llama.cpp

# Verify
llama-server --version

4. Start the LLM Servers (Host — Metal GPU)

The LLM, embedding, and reranker servers run natively on macOS for Metal GPU acceleration. Docker containers connect to them via host.docker.internal.

# Starts LLM (:8080), embedding (:8081), reranker (:8082)
./scripts/start-llama-servers.sh

Wait for all three ports to report healthy (the script auto-checks). First startup loads model weights and takes ~30s.

5. Start Docker Services

# Build and start everything (LightRAG, PostgreSQL, API, Nexus UI)
docker compose up -d --build

# Watch logs
docker compose logs -f

# Check service health
docker compose ps

Services start in dependency order: PostgreSQL → LightRAG → API → Nexus. First build takes a few minutes for the LightRAG image (Python + Rust deps).

6. Verify Everything Works

# LightRAG health
curl -sf http://localhost:9621/health | python3 -m json.tool

# API backend health
curl -sf http://localhost:8000/health | python3 -m json.tool

# LLM (should return model info)
curl -sf http://localhost:8080/health | python3 -m json.tool

# Open Nexus UI (port 3000 in Docker, 5180 in dev mode)
open http://localhost:3000

# Or open LightRAG's built-in WebUI
open http://localhost:9621

7. Process Your First Image

# Via the API (SSE stream — watch real-time progress)
curl -N -X POST http://localhost:8000/images/process \
  -F "file=@photo.jpg"

# Or via the Nexus UI — drag & drop onto the chat input

Stopping Everything

# Stops llama-servers + Docker containers
./scripts/stop-all.sh

# Or just Docker
docker compose down

Key Decisions

Why PostgreSQL?

Persistent, queryable storage for vectors, KV, and doc status. NetworkX for graph storage (AGE extension needed for PG graph — not worth the custom image). HNSW index for fast vector search.

Why host llama-servers?

Apple Silicon Metal GPU acceleration can't run in Docker. llama-server runs natively on the host with full Metal support. LightRAG in Docker connects via host.docker.internal.

Why local build?

docker-compose.yml builds from ./lightrag source rather than pulling ghcr.io/hkuds/lightrag:latest. This lets us modify LightRAG code and rebuild with docker compose up -d --build.

Why VLM-based image ingestion?

LightRAG's /documents/upload only accepts text-based file types (txt, pdf, md, etc.). Image files (.jpg, .png) require the MinerU or Docling parser engines, neither of which is configured in our stack. Instead of deploying an external parser, we send images directly to the VLM (Gemma 4 with mmproj on the same llama-server) to generate a description, then insert that description as text into LightRAG via /documents/text. EXIF metadata (location, date, camera) is injected as separate entity nodes linked to a central Photo node.

Why a combined document instead of separate VLM + metadata docs?

Initially, VLM descriptions and EXIF metadata were inserted as two separate LightRAG documents. This created disconnected subgraphs — "Saint Petersburg" from metadata had no edge to "Backyard" from the VLM description, because LightRAG's entity extraction LLM treated them as unrelated. Combining them into a single document (with EXIF context prepended to the VLM prompt) and injecting EXIF entities as graph nodes with edges to a Photo hub node ensures everything is connected.

Why EXIF entity injection + visual entity linking?

Even in a combined document, LightRAG's entity extraction can't reliably infer that "Saint Petersburg" (from EXIF) and "Backyard" (from VLM) are about the same photo. The reference implementation (memory-search-latest) solves this by programmatically creating Date, Location, and Camera entity nodes with edges (taken_on, taken_at, taken_with) to the photo node, and linking visual entities (Backyard, Person, etc.) back to the photo via appears_in edges. This makes the Photo node the central hub connecting all subgraphs.

Why patch LightRAG's openai.py?

The OpenAI Python SDK (v2.36.0) returns a list instead of CreateEmbeddingResponse when the base_url doesn't end with /v1. Since our EMBEDDING_BINDING_HOST=http://host.docker.internal:8081 lacks the /v1 suffix, every embedding call crashed with 'list' object has no attribute 'data'. The fix normalizes both response shapes and handles the extra nesting in the list case. This is a known SDK compatibility issue with OpenAI-compatible servers.

Why standalone processors?

Image processing modules live in processors/ outside the lightrag/ submodule. They talk to LightRAG via HTTP API, not by patching its source. This means LightRAG updates don't break image processing.

Why Python 3.11 venv?

macOS ships Python 3.9 which has a TensorFlow mutex crash. The project uses a .venv/ with Python 3.11 (installed via Homebrew) where TensorFlow 2.21 + DeepFace work correctly. start-api.sh auto-detects the venv.

Why a persistent job queue?

Image processing involves multiple stages (EXIF extraction → entity creation → VLM description → face recognition) that can take 30+ seconds. A SQLite-backed job queue ensures processing state survives server restarts, prevents duplicate processing, and provides a clean API for tracking progress via SSE. Jobs are created on upload and their status and events are persisted, so clients can reconnect and resume monitoring.

Why nginx reverse proxy?

In production, nginx serves the static SvelteKit build and proxies all API calls — LightRAG, KG API, llama-server, MCP, and WebSocket sync — through a single port (:5180). No CORS issues, no mixed content, clean routing. In dev, Vite's proxy handles the same routes.
